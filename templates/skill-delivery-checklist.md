# 模板：Skill 仓库交付包自检

适用场景：

- 用户要求“改好打包给我”；
- 一次交付多个 Skill / Template / README / SKILL_INDEX 文件；
- 需要生成 zip；
- 需要区分完整文件包与 patch-only 包；
- 需要做最终交付一致性检查。

说明：

- 本模板是低频交付 checklist，不是独立 Skill。
- 使用前应已启用 `skills/00-skill-authoring.md`。
- 如果只是单文件建议或普通评审，不需要读取本模板。

复制使用：

```text
当前任务：对 Skill 仓库变更做交付包自检。

交付方式：
【完整文件包 / patch-only 包 / 仅聊天输出 / 不确定】

声明文件清单：
- 新增：
  - ...
- 修改：
  - ...
- 删除 / 弃用：
  - ...
- 不改但需运行 / 检查：
  - ...

请检查：
1. 声明新增的文件是否实际交付；
2. 声明修改的文件是否实际交付完整替换文件，除非明确标注为 patch-only；
3. 如果是完整文件包，zip 中不能只包含 patch；
4. 如果是 patch-only 包，最终回答和 APPLY.md 必须写清 patch-only；
5. README.md / SKILL_INDEX.md 是否因新增、删除、重命名或弃用而需要同步；
6. scripts/doctor.mjs 是否需要修改；若不需要，说明只需本地运行；
7. 如果影响某个 Skill 的触发方式、关联 Template、输出产物或读取链路，是否需要同步触发提示词；
8. zip 内路径是否与仓库根目录结构一致；
9. 最终回答中的文件清单是否与 zip 实际文件清单一致。

最终输出：
交付一致性检查：
- 声明新增文件：N 个，已交付 N 个
- 声明修改文件：N 个，已交付 N 个 / 或明确 patch-only
- README / SKILL_INDEX：需要更新则已包含；不需要则说明原因
- doctor.mjs：需要修改则已包含；不需要则说明只需运行
- 触发提示词：需要同步 / 不需要同步，并说明原因
- zip 实际文件清单：...
```
