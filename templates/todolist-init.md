# 模板：初始化 TodoList 主线推进账本

适用场景：

- 用户明确需要创建第一版 `todolist.md`；
- 用户要求建立主线推进账本、大方向 TodoList 或防漂移 TodoList；
- 已经有明确的持续任务方向，需要先用审核卡确认主线锚点。

不适用场景：

- 只是一次性问答；
- 只是普通 handoff，且没有已启用的 `todolist.md`；
- 只是讨论长期方向，但还没有要求建立 TodoList。

重要规则：

- 首次初始化默认不要直接生成正式 `todolist.md`。
- 必须先生成「todolist 生成前审核卡」，等待用户审核。
- 审核卡必须是可快速扫读的清单卡，不要写成小型 handoff。
- 用户确认或修正后，才允许生成正式 `todolist.md`。
- 如果用户明确要求直接生成，且输入材料已包含可信的最终目标、当前阶段目标和当前 P0，可以跳过审核卡，但必须在文件中说明依据。

复制到当前窗口使用：

```text
请启用 ChatGPT Skill：目标锚定与 TodoList 初始化。

我想为当前持续任务创建 todolist.md。请先生成「todolist 生成前审核卡」，不要直接生成正式 todolist.md。

要求：
1. 先判断当前任务是否真的需要 todolist.md；如果只是一次性问答或临时任务，请直接说明不需要。
2. 生成前必须先做“目标锚定预检”，复述：最终目标、当前阶段目标、当前 P0。
3. 最终目标、当前阶段目标和当前 P0 必须来自用户明确表达、当前窗口已确认方向，或可信 handoff / todolist.md；不能从 Skill 名称、模板名称、仓库结构、索引、检查脚本、工具链维护上下文里反推。
4. 审核卡必须短，默认不超过 12 行，只包含：
   - 最终目标；
   - 当前阶段目标；
   - 当前 P0；
   - 不进入主线的内容；
   - 待确认候选；
   - 是否建议生成 todolist.md，以及一句话原因；
   - 最后只要求我确认三项锚点是否正确。
5. 审核卡默认使用状态符号：
   - ✓ 已确认 / 方向基本明确；
   - ◐ 当前焦点 / 需要用户确认；
   - ○ 待确认候选；
   - ✕ 不进入主线 / 已排除。
6. 未经我确认，不允许生成正式 todolist.md，也不要生成可下载 todolist.md 文件，除非我明确要求直接生成且材料中三项锚点可信。
7. 如果我修正最终目标、当前阶段目标或当前 P0，必须以我的修正为准。
8. todolist.md 是主线推进账本，不是聊天摘要、命令清单、文件修改清单、检查矩阵或支撑设施维护清单。
9. 大方向 TodoList 必须围绕用户最终目标；文档、索引、模板、脚本、配置、检查命令、测试等支撑设施只能作为 gate、依赖或完成标准，不能默认成为主线目标或 P0。
10. P0 必须是当前阶段直接推进最终目标的事项，或直接阻塞当前阶段交付的事项。
11. 下一步原子任务只能从当前 P0 或当前阶段目标中拆出一个最小动作，而且只能写一个。
12. 对证据不足或用户尚未确认的新方向，放入“待确认 / 候选项”，不要自动升级为 P0。
13. 如果发现目标锚定不清、上一版 TodoList 已经漂移，先指出源头问题，再输出审核卡，不要直接生成修正版 todolist.md。
```

用户确认卡片后，可继续使用：

```text
确认。请按刚才审核卡生成正式 todolist.md，并优先提供可下载文件。
```

如果用户已经提供可信 handoff / todolist.md，并希望跳过审核卡，可使用：

```text
请启用 ChatGPT Skill：目标锚定与 TodoList 初始化。

我已提供可信 handoff / todolist.md，其中包含最终目标、当前阶段目标和当前 P0。请直接生成正式 todolist.md，并在“目标锚定预检”中说明跳过审核 Gate 的依据。
```
