# 模板：初始化 TodoList 主线推进账本

适用场景：

- 当前讨论已经从自由探索进入持续任务；
- 方向基本确定；
- 后续可能需要跨窗口继续推进；
- 需要创建第一版 `todolist.md` 来防止目标漂移。

重要规则：

- 首次初始化时，不要直接生成正式 `todolist.md`。
- 必须先生成「todolist 生成前审核卡」，等待用户审核。
- 审核卡必须是可快速扫读的清单卡，不要写成小型 handoff。
- 用户确认或修正后，才允许生成正式 `todolist.md`。

复制到当前窗口使用：

```text
请启用 ChatGPT Skill：目标锚定与 TodoList 初始化。

当前主题已经从自由讨论进入持续任务。请基于当前窗口已经确认的方向，先生成「todolist 生成前审核卡」，不要直接生成正式 todolist.md。

要求：
1. 先判断当前任务是否真的需要 todolist.md；如果只是一次性问答或临时任务，请直接说明不需要。
2. 生成前必须先做“目标锚定预检”，用两句话复述：最终目标、当前阶段目标。
3. 最终目标必须来自用户当前明确表达或当前窗口已确认方向；不能从 Skill 名称、模板名称、README、SKILL_INDEX、doctor、脚本、仓库维护上下文里反推。
4. 审核卡必须使用清单式结构，便于快速审核：
   - 三个主线锚点：最终目标、当前阶段目标、当前 P0；
   - 不进入主线的内容：支撑设施、候选项、已完成 / 废弃项；
   - 是否建议生成 todolist.md，以及一句话原因；
   - 最后只要求用户确认：最终目标、当前阶段目标、当前 P0 是否正确。
5. 审核卡默认使用状态符号：
   - ✓ 已确认 / 方向基本明确；
   - ◐ 当前焦点 / 需要用户确认；
   - ○ 待确认候选；
   - ✕ 不进入主线 / 已排除。
6. 未经我确认，不允许生成正式 todolist.md，也不要生成可下载 todolist.md 文件。
7. 如果我修正最终目标、当前阶段目标或 P0，必须以我的修正为准。
8. todolist.md 是主线推进账本，不是聊天摘要、命令清单、文件修改清单、覆盖矩阵或仓库治理计划。
9. 大方向 TodoList 必须围绕最终目标；不要把 doctor、README、SKILL_INDEX、模板、脚本维护杂项写成主线目标或 P0。
10. P0 必须是用户最终交付物或直接阻塞该交付物的当前阶段动作。
11. 下一步原子任务只能从当前 P0 或当前阶段目标中拆出一个最小动作，而且只能写一个。
12. 对证据不足或用户尚未确认的新方向，放入“待确认 / 候选项”，不要自动升级为 P0。
13. 如果发现目标锚定不清、上一版 TodoList 已经漂移，先指出源头问题，再输出审核卡，不要直接生成修正版 todolist.md。
```

用户确认卡片后，可继续使用：

```text
确认。请按刚才审核卡生成正式 todolist.md，并优先提供可下载文件。
```
