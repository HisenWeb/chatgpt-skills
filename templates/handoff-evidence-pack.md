# 模板：Handoff Evidence Pack

适用场景：

- 生成 `handoff-current.md` 时，需要同步生成 `handoff-evidence.md`；
- 常规 handoff 或中转蒸馏 handoff 都适用；
- 需要把旧窗口 / 当前窗口中可迁移、可摘录、可说明来源的关键证据完整整理出来；
- 需要在“旧 handoff / 旧 evidence + 当前窗口继续讨论”的场景中做证据基线融合；
- 需要明确哪些证据不能完整写入 Markdown，必须由用户在新窗口重新上传或提供替代证据。

说明：

- 本模板是 `handoff-evidence.md` 的结构模板，不是独立 Skill。
- `handoff-evidence.md` 不应该只是证据摘要；它应尽量保存可复制、可摘录、可迁移的关键文本证据。
- `handoff-evidence.md` 不是原始附件、截图、zip、源码包、视频画面或本地文件的替代品。
- 使用时必须结合 `skills/04-handoff-regular.md` 或 `skills/04b-handoff-distillation.md`。
- 如果涉及 TodoList，TodoList 语义仍以 `skills/04a-goal-todolist.md` 为准。
- 如果缺少关键图片、附件、zip、日志、diff、Prompt 或本地文件，应在生成前或最终回答中提示用户补充；不要默认新增独立补证清单文件。
- 证据融合不是拼接旧文件。旧 evidence 中仍支撑当前结论的证据应保留；当前窗口新增确认应补入；已被用户否定、已失效或不再支撑当前结论的证据应移入“不继承 / 已废弃证据”；新旧证据冲突时必须标注冲突，不要自行调和。

复制使用：

```text
请生成 handoff-evidence.md，结构如下。

# Handoff Evidence Pack

## 1. 本文件用途

说明本文件用于帮助新窗口获得继续工作所需的完整可迁移证据。

必须明确：
- 本文件不是聊天摘要；
- 本文件不是只给结论的证据摘要；
- 本文件应尽量保存可复制、可摘录、可迁移的关键文本证据；
- 本文件支持旧 evidence 的滚动融合，但融合不等于拼接；
- 本文件不是原始附件替代品；
- 图片、截图、zip、源码包、视频画面、本地路径文件等不能自动迁移；
- 无法完整写入本文件的关键外部材料，需要用户重新上传或提供替代证据；
- “handoff-evidence.md 可完整生成”只代表当前可迁移文本证据已完整归档，不代表截图、视频画面、zip、源码包、本地文件等外部原件已经自动携带。

## 2. 证据融合模式与基线审计

### 2.1 融合模式

必须选择并填写一种：

```text
A. 无旧 evidence：基于当前可靠窗口生成证据归档。
B. 有旧 handoff-evidence.md：以旧 evidence 的仍有效证据为基线，滚动融合当前窗口新增证据。
C. 有旧 handoff / handoff zip，但旧 evidence 未完整读取：只能标为待补证，不能声称已融合。
D. 输入是完整旧聊天记录或旧窗口明显污染：本模板不单独解决，应改用中转蒸馏 Skill。
```

### 2.2 基线文件读取情况

| 材料 | 是否存在 | 是否完整读取 | 本次作用 | 未完整读取的影响 |
| --- | --- | --- | --- | --- |
| `handoff-current.md` | 是 / 否 / 不确定 | 是 / 否 / 不适用 | 当前状态基线 / 无 |  |
| `handoff-evidence.md` | 是 / 否 / 不确定 | 是 / 否 / 不适用 | 证据基线 / 无 |  |
| `todolist.md` | 是 / 否 / 不确定 | 是 / 否 / 不适用 | 主线清单基线 / 无 |  |
| 其他附件 / zip / 截图 / 日志 | 是 / 否 / 不确定 | 是 / 否 / 不适用 | 外部证据 / 待补证 |  |

### 2.3 旧证据继承 / 更新 / 废弃表

| 旧证据 | 处理方式 | 原因 | 是否仍支撑当前结论 | 新窗口是否需复核 |
| --- | --- | --- | --- | --- |
|  | 保留 / 更新 / 废弃 / 待补证 |  | 是 / 否 / 部分 | 是 / 否 |

处理规则：
- `保留`：旧证据仍直接支撑当前结论，可继续引用；
- `更新`：旧证据方向仍有效，但需要用当前窗口新增确认修正；
- `废弃`：旧证据已被用户否定、覆盖或不再支撑当前结论；
- `待补证`：旧证据依赖未迁移的图片、zip、源码包、日志、diff 或本地文件。

### 2.4 当前窗口新增证据范围

只列入以下材料：
- 用户明确确认的新结论；
- 用户明确否定的旧判断；
- 新增 TodoList 状态或 checkbox 主线项；
- 新增冲突、证据缺口、待确认问题；
- 已读取的关键文件 / Skill / Template / Prompt 片段。

不要列入：
- 助手跑偏的完整过程；
- 来回纠错的聊天流水；
- 已经被用户否定且对后续无复核价值的中间分析；
- 为了显得完整而拼接的旧 evidence 原文。

## 3. 完整可迁移证据

### 3.1 用户已确认结论与原话

每条证据使用以下格式：

- 证据编号：
- 结论：
- 用户原话 / 完整可迁移摘录：
- 来源：用户明确确认 / 多轮讨论后确认 / 当前窗口明确表达 / 旧 evidence 继承
- 支撑的判断：
- 处理状态：新增 / 继承 / 更新
- 可信度：已确认 / 待复核

要求：
- 能完整摘录的，不要只写摘要；
- 如果只能摘录部分，必须说明未完整携带的部分和影响；
- 旧 evidence 继承来的证据必须说明继承来源，不要伪装成当前窗口新证据。

### 3.2 文件 / Prompt / Skill / 模板证据

每个材料使用以下格式：

- 材料名称：
- 类型：文件 / Prompt / Skill / Template / README / 索引 / 脚本
- 已读取范围：
- 完整可迁移证据：
- 支撑的判断：
- 处理状态：新增 / 继承 / 更新 / 待补证
- 仍需复核：

要求：
- 对关键 Prompt，优先保留完整文本；
- 对关键文件，保留支撑判断所需的完整片段和行号 / 范围；
- 不要只写“已读取某文件，内容合理”。

### 3.3 执行端输出 / 日志 / 报错 / diff / 命令输出

每个材料使用以下格式：

- 原始摘录：
- 关键信息：
- 支撑的判断：
- 不确定点：
- 处理状态：新增 / 继承 / 更新 / 废弃 / 待补证
- 是否已完整携带：是 / 否
- 如果未完整携带，缺失部分：
- 新窗口是否需要重新提供完整日志：是 / 否

要求：
- 关键报错、命令输出、diff 片段应尽量保留原文；
- 如果日志太长，只能摘关键片段时，必须明确这不是完整日志。

### 3.4 图片 / 截图 / 视频画面证据卡

每张关键图片或视频画面使用以下格式：

- 用途：说明它用于证明什么问题

- 不确定点：

- 新窗口是否必须重新上传原图：是 / 否
- 必须重新上传原因：
- 替代方式：

禁止只写“见截图”。
看不清、未识别、未确认的图片细节必须写入“不确定点”。

### 3.5 不继承 / 已废弃证据

用于短记录被用户否定、已失效或不应进入新窗口主线的材料。

| 材料 / 判断 | 废弃原因 | 后续处理 |
| --- | --- | --- |
|  | 用户否定 / 已被新结论覆盖 / 助手误判 / 无后续价值 | 不继承 / 仅作反例 / 需要重新验证 |

要求：
- 只记录对新窗口有防漂移价值的废弃项；
- 不展开完整争论过程；
- 不把废弃项写入当前状态、已确认结论或 TodoList。

- 类型：截图 / 图片 / zip / md / 源码包 / 日志 / diff / 本地文件 / GitHub 文件 / File Library 文件

## 5. 未完整携带 / 需要外部补充的证据

用表格列出：

| 材料 | 类型 | 是否必须 | 原因 | 不提供的影响 | 替代方式 |
| --- | --- | --- | --- | --- | --- |
|  | 截图 / 附件 / zip / 日志 / diff / Prompt / 本地文件 / 其他 | 必须 / 建议 |  |  |  |

必须覆盖：
- 图片 / 截图原图；
- zip / 源码包 / 附件原件；
- 旧窗口上传过但新窗口未上传的文件；
- 用户本地路径指向的文件；
- 未完整读取过的文件；
- 执行端日志、命令输出、diff、测试结果；
- 需要视觉复核、代码复核或重新执行才能确认的材料；
- 旧 evidence 中标注但本次未完整迁移的材料。

如果没有缺失材料，必须写：

```text
当前未发现必须额外补充的外部证据。
```

## 7. 不需要携带 / 禁止继承的材料

列出不需要迁移或禁止继承的材料类型，例如：

- 已解决且无后续影响的中间日志；
- 已废弃方案的完整过程；
- 与当前阶段无关的旧聊天记录；
- 重复解释；
- 情绪化争执；
- 不影响下一步判断的临时尝试；
- 助手跑偏但已被用户纠正、且没有复核价值的分析过程；

## 8. 新窗口读取顺序建议

根据实际输出列出：

1. 先读 `handoff-current.md`；
2. 再读 `handoff-evidence.md`；
3. 如果存在 `todolist.md`，再读 `todolist.md`；
4. 如果发现 `handoff-evidence.md` 标注了“未完整携带 / 需要外部补充”的关键证据，先提醒用户补充，不要把缺失证据写成已确认事实；
5. 如果 `handoff-current.md`、`handoff-evidence.md`、`todolist.md` 之间冲突，先指出冲突，不要自行合并；
6. 如果新窗口需要继续生成下一轮 handoff，必须以本轮 handoff-current / handoff-evidence / todolist 为基线滚动更新，不要重新总结聊天流水。
