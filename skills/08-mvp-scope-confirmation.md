# 08-mvp-scope-confirmation — MVP 业务范围说明

## 触发场景

当用户已经完成一轮自由需求讨论、头脑风暴、旧窗口交接或需求草稿整理，但还没有形成可用于生成原型的清晰业务说明时，启用本 Skill。

典型触发：

- 用户说“先收敛一下 MVP”“先把业务范围理清楚”“生成原型前先确认范围”；
- 用户提供了一批需求材料、想法、竞品参考、旧窗口讨论或业务草稿，需要整理成原型输入；
- 用户准备进入低保真页面板或可点击原型，但业务场景、页面含义、用户流程或关键状态仍不清楚；
- 当前讨论出现业务概念混杂、页面职责不清、流程断点、状态归属不明的问题；
- 用户希望把复杂想法压缩成一份面向后续原型生成的业务说明。

## 不触发场景

以下情况不要启用本 Skill：

- 用户还在自由探索方向，目标用户、业务场景或核心问题完全没形成；
- 用户已经给出清晰页面说明，只需要直接生成低保真页面板；
- 用户已经给出清晰主流程、页面关系和关键交互，只需要生成可点击原型；
- 用户正在修已有 HTML 原型的细节、样式、Bug 或跳转；
- 用户只是要求图片、高保真设计、生产代码、技术方案、开发计划或执行端流程；
- 用户正在维护 Skill 仓库本身，此时优先启用 `skills/00-skill-authoring.md`。

如果材料已经足够进入后续阶段，不要为了形式再产出一份冗长范围文档；只需简短确认可进入 `09` 或 `10`。

## 核心定位

本 Skill 的定位是：把需求讨论后的业务材料整理成“后续原型能看懂、能生成、能评审”的业务范围说明。

它不是总控决策文档，不是开发指令，不是完整 PRD，不负责规定技术方案，也不替后续执行端做实现判断。

核心输出是：

```text
MVP 业务范围说明
```

这份说明只记录业务和原型相关事实：业务场景、用户流程、页面含义、关键状态、业务规则和仍需确认的问题。

## 工作原则

### 0. 正式文档必须在用户确认后生成

本 Skill 不应一启用就直接生成正式交付文件。

流程必须是：

```text
先输出 MVP 业务范围草案 / 确认卡
→ 用户明确确认或修正
→ 才生成 mvp-business-scope.md
```

未确认前只能输出草案、确认卡、待确认问题或下一步建议；不能把内容固化成 `mvp-business-scope.md`。

### 1. 先判断是否已经聊到可收口

本 Skill 的最佳调用点是：已经聊出一批业务材料，但还没有整理成可原型化输入。

不要太早调用：如果用户还在比较方向、目标用户不明、业务场景不明，应继续自由讨论。

不要太晚调用：如果页面说明、用户流程、关键状态和原型形态已经清楚，可以直接进入页面板或可点击原型。

### 2. 只管业务和原型，不管开发执行

输出只围绕后续原型需要理解的内容：

- 这个原型表达什么业务场景；
- 用户在场景里想完成什么；
- 原型需要呈现哪些页面或页面组；
- 用户从开始到结束的大致业务流程；
- 页面里的业务术语、数据和状态分别是什么意思；
- 哪些问题会影响后续原型表达。

不要输出：技术栈、项目路径、启动命令、文件映射、API 设计、数据库设计、开发计划、执行端规则或具体工具流程。

### 3. 用“原型表达 / 原型未表达”替代总控式范围命令

本 Skill 不用“必做 / 不做 / 候选”这类总控口吻。

更合适的表达是：

- 原型表达的业务场景；
- 原型覆盖的用户流程；
- 原型涉及的页面和状态；
- 原型暂未表达、但可能影响理解的问题；
- 原型之外的背景信息是否只是参考。

这样后续执行端拿到的是业务理解材料，而不是被网页端提前规定的开发命令。

### 4. 外部材料只作为输入，不绑定具体工具

可以接收任意外部需求、设计、规格、聊天、截图或交接材料，但不要在 Skill 中绑定任何具体工具或工作流。

处理外部材料时必须区分：

- 已经被用户确认的业务结论；
- 只是参考的历史材料；
- 与当前原型表达无关的噪音；
- 仍会影响页面或流程理解的问题。

不要假设旧窗口附件、图片、zip、源码包或聊天记录已经自动迁移；只有当前可见材料才能作为事实。

### 5. 输出必须能支撑页面板和可点击原型

业务范围说明必须足够具体，能让下一阶段回答这些问题：

- 页面板需要展示哪些页面；
- 每个页面表达什么业务含义；
- 哪些状态归属于哪个页面；
- 可点击原型需要表达哪条用户流程；
- 哪些交互结果是业务上必须表达的；
- 哪些业务术语需要统一。

不要只写抽象愿景，也不要把所有讨论过程搬进文档。

### 6. 不把缺失信息补成事实

如果材料不足，应明确标注为“仍需业务确认”。

允许基于材料给出建议整理，但必须区分：

- 已确认业务结论；
- 基于材料的整理建议；
- 仍需用户确认的问题。

不要为了让文档完整而编造页面、状态、业务规则或用户流程。

## 输出要求

### 1. 未确认前：输出 MVP 业务范围草案 / 确认卡

未得到用户明确确认前，只输出草案，不生成正式文件。

推荐结构：

```text
MVP 业务范围草案

1. 原型表达的业务场景
- ...

2. 用户场景
- 用户是谁 / 在什么情况下使用 / 想完成什么

3. 核心业务流程
- 用户从哪里开始 → 经过哪些业务步骤 → 到哪里结束

4. 页面草案
- 页面 A：业务用途 ...
- 页面 B：业务用途 ...

5. 页面状态草案
- 页面 A：默认态 / 空状态 / 完成态 / 其他必要状态
- 页面 B：...

6. 关键业务规则
- 术语、数量、记录口径、结果口径等

7. 术语与数据含义
- 业务词：含义
- 页面数据：代表什么

8. 仍需业务确认的问题
- 只列会影响原型表达的问题

9. 建议下一步
- 进入 HTML 原型生成的页面板模式 / 进入 HTML 原型生成的可点击模式 / 继续补业务信息
```

### 2. 下一步判定

草案结尾给出一个简短判定：

- `READY_FOR_PAGE_BOARD`：业务页面和状态足够清楚，可以进入 HTML 原型生成的页面板模式；
- `READY_FOR_CLICKABLE_PROTOTYPE`：业务流程和关键交互足够清楚，可以进入 HTML 原型生成的可点击模式；
- `NEEDS_BUSINESS_CLARIFICATION`：仍缺少会影响原型表达的业务信息；
- `TOO_BROAD_FOR_PROTOTYPE_BRIEF`：材料太散，需要继续压缩业务场景；
- `NOT_A_PROTOTYPE_TASK`：当前不是原型前业务范围说明任务。

### 3. 用户确认后：生成正式文件 `mvp-business-scope.md`

只有用户明确确认业务范围草案后，才生成正式文件：

```text
mvp-business-scope.md
```

正式文件使用以下结构：

```text
# MVP Business Scope

## 1. 原型表达的业务场景
## 2. 用户场景
## 3. 核心业务流程
## 4. 页面草案
## 5. 页面状态草案
## 6. 关键业务规则
## 7. 术语与数据含义
## 8. 待确认业务问题
## 9. 下一步建议
```

`mvp-business-scope.md` 是后续 `09-html-prototype-generator.md` 和 `10-prototype-freeze-gate.md` 的业务范围源。

### 4. 与后续 Skill 的衔接

如果可以进入下一阶段，按实际需求建议：

- 需要审页面结构、信息密度、页面状态归属：进入 `skills/09-html-prototype-generator.md` 的页面板模式；
- 需要验证业务流程、按钮结果和页面流转：进入 `skills/09-html-prototype-generator.md` 的可点击原型模式；
- 已有 `mvp-business-scope.md`、`prototype-delivery.md` 和最终 `prototype/`，需要冻结评审与打包：进入 `skills/10-prototype-freeze-gate.md`。

不要在本 Skill 中直接生成 HTML 原型或面向执行端的开发说明。

## 禁止事项

- 禁止把业务范围说明写成完整 PRD。
- 禁止使用“必做 / 不做 / 候选”这类总控式开发口吻。
- 禁止替用户或后续执行端做技术方案、开发计划或实现决策。
- 禁止绑定任何具体执行工具、规格框架或 agent 工作流。
- 禁止把用户未确认的假设写成事实。
- 禁止把历史聊天、截图、日志、diff、旧材料清单原样转交给后续阶段。
- 禁止在业务范围不清时直接生成 HTML 原型。
- 禁止为了完整而补大量页面、角色、权限、后台、商业化或工程能力。
- 禁止把支撑设施、流程 gate、工程实现细节当成业务主线。
- 禁止把具体项目名、路径、工具、模型、执行端或某次窗口事故写成通用规则。
