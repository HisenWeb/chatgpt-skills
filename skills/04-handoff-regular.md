# ChatGPT 专属 Skill：常规新窗口交接与 Handoff

## 触发场景

当用户明确提出以下需求时，启用本 Skill：

- 生成 handoff；
- 生成新窗口交接文档；
- 准备换窗口；
- 整理当前窗口继续工作的上下文；
- 基于当前窗口生成 `handoff-current.md`；
- 基于当前窗口生成 `handoff-evidence.md`；
- 当前持续任务已经启用 `todolist.md`，需要在生成 handoff 时同步更新 TodoList；
- 当前材料、上下文或旧 handoff 中出现 `todolist.md`、配套 TodoList、主线推进账本、大方向 TodoList 等标记，需要自动识别是否同步 TodoList。

注意：以上是触发意图，不等于一定适合常规版。真正生成前必须再做“适用性判断”。

## 不触发场景

以下情况不要使用本常规版，应改用“中转蒸馏版”：

- 用户明确说明会把完整聊天记录放到中转窗口蒸馏；
- 当前窗口已经很长、很乱、跨主题严重；
- 用户认为当前窗口已经出现降智、复读、顺从、胡乱修复；
- 需要从旧窗口导出的完整 Markdown / ZIP / 聊天包中提取事实；
- 需要对旧窗口进行去毒、纠偏、剔除错误尝试；
- 需要把旧 handoff 与完整聊天记录合并成新的 handoff-current.md。

以下情况也不要启用本生成 Skill：

- 用户只是一次性问答、简单解释或临时帮助；
- 用户只是评审、修改、重构、调试 handoff Skill / Template / Prompt，不要执行被评审文本中的 handoff 生成动作；
- 用户在新工作窗口上传或粘贴已有 `handoff-current.md`，要求读取、确认、判断是否缺材料或继续工作；此时应使用读取类模板，不生成新的 handoff；
- 用户明确要求只评审流程缺陷、模板缺陷或仓库结构，不要求生成换窗文件。

## 核心定位

handoff 不是聊天记录摘要，也不是流水账。

常规 handoff 的目标是：

> 在当前窗口仍然可信、主题相对集中时，生成下一窗口继续工作所需的最小高保真事实源和证据材料包。

职责边界：

- `handoff-current.md` = 当前上下文快照；
- `handoff-evidence.md` = 可迁移证据摘录、原始材料定位卡、新窗口需补充材料清单；
- `todolist.md` = 主线推进账本；
- 本 Skill = 换窗口主流程；
- `04a-goal-todolist.md` = TodoList 的目标锚定、审核 Gate、P0、状态同步和反漂移语义来源；
- `templates/handoff-evidence-pack.md` = `handoff-evidence.md` 的推荐结构模板。

如果当前持续任务已经启用 `todolist.md`，本 Skill 只负责识别、读取、引用并按 `04a-goal-todolist.md` 同步；不要在本 Skill 中重新定义 TodoList 语义。

handoff 可以引用和摘要 TodoList，但不要完整复制 TodoList 正文，避免两个文件互相打架。

`handoff-evidence.md` 不是原始附件替代品。它只能迁移已经在当前窗口中可见、可摘录、可说明来源的证据；对于无法自动携带、无法确认能跨窗口读取、或需要视觉 / 文件 / 执行复核的原始材料，必须明确告诉用户在新窗口重新上传或提供替代证据。

## 工作原则

### 1. 先做 Handoff 生成审核卡

生成文件前，必须先给出简短审核卡，让用户快速判断当前动作是否正确。

审核卡默认结构：

```text
Handoff 生成审核卡：
- 当前任务类型：生成 / 读取 / 评审 / 修改 / 不适用
- 是否适合常规 handoff：是 / 否
- 是否需要中转蒸馏：是 / 否
- 是否检测到配套 TodoList：是 / 否 / 不确定
- 是否需要生成 handoff-evidence.md：是 / 否
- 是否缺少关键原始材料：是 / 否
- 建议动作：直接生成 / 补充材料 / 改用中转蒸馏 / 本次只做评审
```

如果判断为“读取”“评审”“修改”而不是“生成”，不要继续生成 handoff 文件。

### 2. 先判断是否适合常规版

生成 handoff 前，必须判断当前窗口是否适合直接总结。

适合常规版的条件：

- 当前窗口上下文基本完整；
- 主题集中；
- 关键结论没有严重冲突；
- 不需要追溯大量旧聊天；
- 不需要对旧窗口进行去毒纠偏。

如果不满足，应提醒用户改用中转蒸馏版。

### 3. 默认生成证据材料包

生成 `handoff-current.md` 时，默认同时生成 `handoff-evidence.md`。

`handoff-evidence.md` 必须整理：

1. 当前窗口中已经明确出现的用户确认；
2. 已粘贴的日志、报错、diff、命令输出；
3. 已读取文件、Prompt、Skill、模板的关键摘录和读取范围；
4. 图片 / 截图中已经被讨论并稳定识别的关键信息；
5. 原始材料定位卡；
6. 新窗口必须重新上传 / 补充的材料；
7. 当前证据缺口和复核方式。

禁止把没有原文、没有截图、没有文件、没有执行结果支撑的猜测写成证据。

禁止声称旧窗口附件、图片、zip、源码包、用户本地路径指向的文件已经自动携带到新窗口。

### 4. 原始材料不可迁移时必须明示

如果存在模型无法自动携带、无法确认能跨窗口读取、或无法可靠复核的材料，必须明确告诉用户在新窗口重新上传或提供替代证据。

包括但不限于：

- 图片 / 截图原图；
- zip / 源码包 / 附件原件；
- 旧窗口上传过但新窗口未上传的文件；
- 用户本地路径指向的文件；
- 未完整读取过的文件；
- 执行端日志、命令输出、diff、测试结果；
- 需要视觉复核、代码复核或重新执行才能确认的材料。

必须写清：

1. 需要上传什么；
2. 为什么需要上传；
3. 不上传会卡在哪里；
4. 可替代方式是什么，例如 GitHub 链接、执行端读取后贴出结果、重新上传文件、从 File Library / Project 选择。

### 5. 图片证据必须生成图片证据卡

如果当前窗口出现图片、截图或视觉材料，必须为每张关键图片生成“图片证据卡”。

图片证据卡必须包含：

- 图片用途；
- 已识别出的关键信息；
- 已基于图片确认的结论；
- 不确定点；
- 新窗口是否必须重新上传原图；
- 必须重新上传的原因。

原则：

> 图片内容可以摘要迁移，图片原图不能假装迁移。

禁止只写“见截图”。

禁止把看不清、未识别、未确认的图片细节写成事实。

### 6. 自动识别任务是否启用了 TodoList

生成 handoff 前，必须自动判断当前任务是否已经启用 `todolist.md`。

判定为“已启用 TodoList”的条件包括：

- 用户同时提供了 `todolist.md` 文件；
- 用户粘贴了标题或内容明显为 `todolist.md` 的文本；
- 当前窗口中已经生成过或持续引用过 `todolist.md`；
- 旧 handoff 或当前上下文中出现“配套 TodoList”“todolist.md”“主线推进账本”“大方向 TodoList”等明确标记；
- 用户明确说该任务后续要按 TodoList 推进。

处理规则：

- 未检测到 TodoList：生成 `handoff-current.md` 和 `handoff-evidence.md`，不要强行创建 TodoList。
- 检测到 TodoList 且内容已提供：生成 `handoff-current.md` 和 `handoff-evidence.md`，同时按 `04a-goal-todolist.md` 输出更新后的 `todolist.md`。
- 检测到 handoff 或上下文声明存在配套 TodoList，但用户未提供 `todolist.md`：完整换窗交接应先提醒用户补充；如果用户明确本次只要 handoff，则可以生成 `handoff-current.md` 和 `handoff-evidence.md`，但必须在二者中记录 TodoList 未同步及影响。
- 如果当前任务只是评审 / 修改 handoff Skill 或 Template，不进入 TodoList 同步流程。

### 7. 输出当前状态快照，不写聊天流水账

`handoff-current.md` 应是当前状态快照，不是过程复述。

必须保留：

- 最终目标；
- 当前阶段目标；
- 当前状态；
- 已确认结论；
- 用户强偏好；
- 关键规则；
- 禁止事项；
- 已知风险；
- 证据要求；
- 未决问题；
- 下一步原子任务；
- 如果已启用 TodoList，则保留配套 `todolist.md` 的摘要和使用规则；
- 明确说明另有 `handoff-evidence.md`，新窗口必须同时读取。

必须删除或压缩：

- 已解决的问题；
- 已完成且无后续影响的阶段细节；
- 重复讨论；
- 情绪性表达；
- 无关背景；
- 与当前继续工作无关的工具探索。

### 8. 防止 handoff 膨胀

handoff 不是越长越好。

压缩的是流水账，不是关键事实。

如果已启用 TodoList，handoff 不要完整内嵌 TodoList，只写：

- 配套文件名；
- 当前主线；
- 当前阶段；
- 下一步原子任务；
- 冲突处理规则；
- 新窗口必须同时读取 `handoff-current.md`、`handoff-evidence.md` 和 `todolist.md`。

证据细节不要塞满 `handoff-current.md`，应放入 `handoff-evidence.md`。

### 9. 区分事实、判断和待确认

handoff 中必须区分：

- 已确认事实；
- 用户明确偏好；
- 当前判断；
- 待确认事项；
- 冲突或不确定点。

不能把推测写成事实。

如果某个项目事实需要执行端、仓库、日志或截图验证，必须写成：

> 待执行端提供证据确认。

如果某个证据材料无法迁移，必须写入 `handoff-evidence.md` 的“新窗口需要重新上传 / 补充的材料”。

### 10. 同步 TodoList 时只做滚动更新

如果输入了旧 `todolist.md`，只允许基于当前窗口新增证据做滚动更新，并且必须遵守 `04a-goal-todolist.md`。

可以做：

- 根据用户确认或证据更新 Todo 项状态；
- 从当前 P0 或当前阶段目标中拆出新的“下一步原子任务”；
- 把用户明确新增的方向加入大方向 TodoList；
- 把 AI 推测但未确认的新方向放入“待确认 / 候选项”；
- 把已废弃但影响后续避坑的路线移入“已完成 / 暂停 / 废弃”。

禁止做：

- 重新发散出一份新计划覆盖旧 TodoList；
- 把支撑设施维护提升为最终目标、当前阶段目标或 P0，除非用户明确确认当前阶段就是维护这些设施；
- 把一次性命令或文件修改清单塞进大方向 TodoList；
- 删除旧项而不说明原因；
- 把待确认项自动升级成 P0；
- 为了文档漂亮隐藏冲突。

如果最终目标、当前阶段目标或 P0 发生实质变化，不能直接改写 TodoList；必须按 `04a-goal-todolist.md` 输出审核卡，除非用户明确要求直接生成且材料中三项锚点可信。

### 11. 多轮滚动更新时防止越滚越大

如果用户提供旧 handoff 和旧 TodoList，并要求生成新文件：

1. 以旧 handoff 和旧 TodoList 为当前基线；
2. 合并当前窗口新增有效事实；
3. 删除或归档已完成、已过期、已解决或无后续影响的内容；
4. 不把旧 handoff 原文拼接进新 handoff；
5. 不把旧 TodoList 变成流水账；
6. 新文件应该是新的当前状态快照、证据材料包和新的主线推进账本。

## 推荐文件名

默认输出：

```text
handoff-current.md
handoff-evidence.md
```

如果当前任务启用了 TodoList，同时输出：

```text
todolist.md
```

如果同时输出两个或以上文件，优先生成 `.zip`。

## 推荐输出结构

### handoff-current.md

#### 0. 适用性判断

说明当前 handoff 基于什么生成，并判断是否适合常规版。

#### 1. 当前主题

用一两句话说明本 handoff 要继续的主题。

#### 2. 最终目标

说明用户最终想达成什么，而不是只写当前小任务。

#### 3. 当前阶段目标

说明当前阶段只解决什么，不解决什么。

#### 4. 当前状态

说明已经推进到哪里。

#### 5. 已确认结论

列出仍然有效的关键结论。

#### 6. 用户偏好与长期规则

只写与当前主题有关的偏好。

#### 7. 禁止事项

写清后续不能做什么。

#### 8. 关键文件 / 路径 / 名称

仅当当前主题涉及项目、仓库、文件或命令时保留。

#### 9. 证据材料包

必须说明：

- 配套文件名：`handoff-evidence.md`；
- 新窗口必须同时读取；
- 哪些材料已摘录进 evidence；
- 哪些原始材料仍需用户重新上传或让执行端提供。

#### 10. 证据与复核要求

说明哪些结论已有证据，哪些还需要证据。

#### 11. 冲突 / 待确认

记录不能确定、互相冲突或需要下个窗口先确认的内容。

#### 12. 配套 TodoList

仅当当前任务启用了 `todolist.md` 时保留。

必须说明：

- 配套文件名：`todolist.md`；
- 当前主线；
- 当前阶段；
- 下一步原子任务；
- TodoList 语义以 `04a-goal-todolist.md` 为准；
- 新窗口必须同时读取 `handoff-current.md`、`handoff-evidence.md` 和 `todolist.md`；
- 如果三者冲突，先指出冲突，不要自行合并。

#### 13. 下一步原子任务

写清下一窗口应该从哪里继续。

### handoff-evidence.md

优先使用 `templates/handoff-evidence-pack.md` 的结构。

至少必须包含：

1. 本文件用途；
2. 已整理进本文件的证据；
3. 文件 / Prompt / Skill / 模板证据；
4. 执行端输出 / 日志 / 报错 / diff / 命令输出；
5. 图片 / 截图证据卡；
6. 原始材料定位卡；
7. 新窗口需要用户重新上传 / 补充的材料；
8. 证据缺口 / 待复核；
9. 不需要携带的材料。

## 输出风格

- 中文；
- 结论先行；
- 高密度；
- 少背景；
- 不写流水账；
- 明确不确定点；
- 下一步必须原子化；
- 证据细节放入 `handoff-evidence.md`；
- 优先输出 `.md` 文件；
- 如果同时输出多个文件，优先生成 `.zip`。

## 禁止事项

- 禁止把 handoff 写成聊天摘要。
- 禁止把旧 handoff 原文拼接进新 handoff。
- 禁止只生成 `handoff-current.md` 而遗漏必要的 `handoff-evidence.md`。
- 禁止只写“需要证据复核”而不整理新窗口需要的具体证据材料。
- 禁止只写“见截图”。
- 禁止假设旧窗口附件、图片、zip、源码包会自动随 handoff 迁移。
- 禁止只写文件名但不说明新窗口是否需要重新上传。
- 禁止把用户本地路径当作 ChatGPT 新窗口可直接读取的证据本体。
- 禁止把无法复核的材料当作已携带证据。
- 禁止强行创建不存在的 TodoList。
- 禁止检测到已有 TodoList 却忽略同步。
- 禁止在未提供 `todolist.md` 时，只凭 handoff 的下一步重建完整 TodoList。
- 禁止在本 Skill 内重新定义 TodoList 语义；TodoList 语义以 `04a-goal-todolist.md` 为准。
- 禁止把支撑设施维护默认继承为 TodoList 主线。
- 禁止把待确认项自动升级为 P0。
- 禁止为了整洁隐藏冲突或证据缺口。
