# ChatGPT 专属 Skill：常规新窗口交接与 Handoff

## 触发场景

当用户提出以下需求时，启用本 Skill：

- 生成 handoff；
- 生成新窗口交接文档；
- 准备换窗口；
- 整理当前窗口继续工作的上下文；
- 当前窗口主题相对集中，关键结论主要在当前窗口内；
- 当前窗口仍然可靠，不需要从完整旧聊天记录进行中转蒸馏。

## 不触发场景

以下情况不要使用本常规版，应改用“中转蒸馏版”：

- 用户明确说明会把完整聊天记录放到中转窗口蒸馏；
- 当前窗口已经很长、很乱、跨主题严重；
- 用户认为当前窗口已经出现降智、复读、顺从、胡乱修复；
- 需要从旧窗口导出的完整 Markdown / ZIP / 聊天包中提取事实；
- 需要对旧窗口进行去毒、纠偏、剔除错误尝试；
- 需要把旧 handoff 与完整聊天记录合并成新的 handoff-current.md。

## 核心定位

handoff 不是聊天记录摘要，也不是流水账。

常规 handoff 的目标是：

> 在当前窗口仍然可信、主题相对集中时，生成下一窗口继续工作所需的最小高保真事实源。

它必须帮助新窗口快速恢复：

1. 当前目标；
2. 当前状态；
3. 已确认结论；
4. 用户偏好和规则；
5. 禁止事项；
6. 证据要求；
7. 未决问题；
8. 下一步动作。

## 工作原则

### 1. 先判断是否适合常规版

生成 handoff 前，必须先判断当前窗口是否适合直接总结。

适合常规版的条件：

- 当前窗口上下文基本完整；
- 主题集中；
- 关键结论没有严重冲突；
- 不需要追溯大量旧聊天；
- 不需要对旧窗口进行去毒纠偏。

如果不满足，应提醒用户改用中转蒸馏版。

### 2. 输出当前状态快照，不写聊天流水账

handoff 应是当前状态快照，不是过程复述。

必须保留：

- 仍有效的目标；
- 当前阶段；
- 已确认结论；
- 用户强偏好；
- 关键规则；
- 禁止事项；
- 已知风险；
- 证据要求；
- 未决问题；
- 下一步动作。

必须删除或压缩：

- 已解决的问题；
- 已完成且无后续影响的阶段细节；
- 重复讨论；
- 情绪性表达；
- 无关背景；
- 与当前继续工作无关的工具探索。

### 3. 防止 handoff 膨胀

handoff 不是越长越好。

但也不能只写轻量方向总结。

压缩的是流水账，不是关键事实。

必须保留对后续继续工作有影响的：

- 文件路径；
- 命令名；
- 工具名；
- 协议字段；
- 关键配置语义；
- 阶段 gate；
- 验收标准；
- 证据要求；
- 用户明确禁止事项。

### 4. 区分事实、判断和待确认

handoff 中必须区分：

- 已确认事实；
- 用户明确偏好；
- 当前判断；
- 待确认事项；
- 冲突或不确定点。

不能把推测写成事实。

如果某个项目事实需要执行端、仓库、日志或截图验证，必须写成：

> 待执行端提供证据确认。

### 5. 多轮滚动更新时防止越滚越大

如果用户提供旧 handoff，并要求生成新的 handoff-current.md：

1. 以旧 handoff 为当前基线；
2. 合并当前窗口新增有效事实；
3. 删除已完成、已过期、已解决或无后续影响的内容；
4. 不把旧 handoff 原文拼接进新 handoff；
5. 新 handoff 应该是新的当前状态快照。

## 推荐文件名

默认输出：

`handoff-current.md`

如果用户要求按主题命名，可使用：

- `handoff-cc-switch.md`
- `handoff-open-design-workflow.md`
- `handoff-v3s.md`

滚动更新时仍建议使用：

`handoff-current.md`

不要默认生成：

- `handoff-v1.md`
- `handoff-v2.md`
- `handoff-final-final.md`

除非用户明确要求保留历史版本。

## 推荐输出结构

### 0. 适用性判断

说明当前 handoff 基于什么生成：

- 当前窗口；
- 旧 handoff + 当前窗口；
- 用户粘贴材料；
- 用户上传文件。

同时说明可靠性：

- 是否上下文完整；
- 是否有截断风险；
- 是否有证据缺口；
- 是否需要执行端补充材料。

### 1. 当前主题

用一两句话说明本 handoff 要继续的主题。

### 2. 当前目标

说明用户最终想达成什么，而不是只写当前小任务。

### 3. 当前状态

说明已经推进到哪里。

如果是阶段化任务，应写清：

- 当前阶段；
- 已完成阶段；
- 未完成阶段；
- 阻塞点；
- 下一阶段进入条件。

### 4. 已确认结论

列出仍然有效的关键结论。

要求：

- 结论可复用；
- 不写无关讨论；
- 不写已废弃的中间想法，除非它会影响后续避坑。

### 5. 用户偏好与长期规则

只写与当前主题有关的偏好。

例如：

- 中文表达；
- 结论先行；
- 最小改动；
- 阶段化执行；
- 证据复核；
- 不要 Yes-man；
- 不要工具地狱；
- 不要默认写死具体执行工具；
- 长文档优先生成 `.md` 文件。

### 6. 禁止事项

写清后续不能做什么。

例如：

- 不要把临时验证正式化；
- 不要未审计入口就改 patch；
- 不要把旧 handoff 原文拼接进新 handoff；
- 不要凭空假设已经读过项目；
- 不要扩展范围；
- 不要强行引入新工具或新流程。

### 7. 关键文件 / 路径 / 名称

仅当当前主题涉及项目、仓库、文件或命令时保留。

包括：

- 仓库名；
- 本地路径；
- 文件路径；
- 命令；
- 分支；
- 版本；
- 重要配置文件；
- 关键协议字段。

如果不涉及项目文件，可以省略。

### 8. 证据与复核要求

说明哪些结论已有证据，哪些还需要证据。

可记录证据类型：

- diff；
- 日志；
- 命令输出；
- 截图；
- 测试结果；
- 文件内容；
- 用户确认。

### 9. 冲突 / 待确认

记录不能确定、互相冲突或需要下个窗口先确认的内容。

不能为了文档漂亮而隐藏不确定性。

### 10. 下一步建议

写清下一窗口应该从哪里继续。

要求：

- 具体；
- 可执行；
- 不要开太多分支；
- 不要把已经完成的事情再做一遍。

### 11. 新窗口启动提示

最后提供一段可直接复制到新窗口的启动提示。

启动提示应简洁，只负责让新窗口读取 handoff 并确认理解。

不要把“生成 handoff 的规则”塞进启动提示。

## 新窗口启动提示模板

### 剪贴板版

适用于用户直接把 handoff 内容粘贴到新窗口。

```text
我们继续上一个窗口的主题。下面是 handoff-current.md，请先阅读并确认你理解当前目标、状态、规则、禁止事项、未决问题和下一步。

要求：
1. 不要重新发散。
2. 不要重复已经完成的工作。
3. 如果发现 handoff 内部冲突或证据不足，先指出。
4. 后续回答遵守 handoff 中的用户偏好和工作边界。

handoff-current.md 内容如下：
【粘贴 handoff】
```

### 文件版

适用于用户上传 handoff 文件到新窗口。

```text
我们继续上一个窗口的主题。我已上传 handoff-current.md。

请先读取该文件，并用简短要点确认：
1. 当前目标；
2. 当前状态；
3. 必须遵守的规则；
4. 禁止事项；
5. 未决问题；
6. 下一步建议。

确认后再继续工作，不要重新发散。
```

## 输出风格

- 中文；
- 结构清晰；
- 结论先行；
- 少写废话；
- 保留关键细节；
- 不写流水账；
- 明确不确定点；
- 优先输出 `.md` 文件；
- 用户要直接复制时，再输出正文。

## 禁止事项

- 禁止把 handoff 写成聊天摘要。
- 禁止把 handoff 写成流水账。
- 禁止只给轻量方向总结。
- 禁止遗漏关键规则、边界、证据要求和待确认项。
- 禁止多轮 handoff 直接拼接，导致越滚越大。
- 禁止把猜测写成事实。
- 禁止假装已经读过未提供的项目、仓库、日志或文件。
- 禁止在 handoff 中引入用户没有确认的新工具路线。
- 禁止把“读取 handoff 的启动提示”和“生成 handoff 的完整规则”混在一起。
- 禁止在复杂 handoff 中使用容易复制损坏的嵌套 Markdown 大代码块。
