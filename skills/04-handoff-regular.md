# ChatGPT 专属 Skill：常规新窗口交接与 Handoff

## 触发场景

当用户提出以下需求时，启用本 Skill：

- 生成 handoff；
- 生成新窗口交接文档；
- 准备换窗口；
- 整理当前窗口继续工作的上下文；
- 当前窗口主题相对集中，关键结论主要在当前窗口内；
- 当前窗口仍然可靠，不需要从完整旧聊天记录进行中转蒸馏；
- 当前持续任务已经启用 `todolist.md`，需要在生成 handoff 时同步更新 TodoList。

## 不触发场景

以下情况不要使用本常规版，应改用“中转蒸馏版”：

- 用户明确说明会把完整聊天记录放到中转窗口蒸馏；
- 当前窗口已经很长、很乱、跨主题严重；
- 用户认为当前窗口已经出现降智、复读、顺从、胡乱修复；
- 需要从旧窗口导出的完整 Markdown / ZIP / 聊天包中提取事实；
- 需要对旧窗口进行去毒、纠偏、剔除错误尝试；
- 需要把旧 handoff 与完整聊天记录合并成新的 handoff-current.md。

如果当前任务只是一次性问答、简单解释或临时帮助，不需要生成 handoff，也不需要 TodoList。

## 核心定位

handoff 不是聊天记录摘要，也不是流水账。

常规 handoff 的目标是：

> 在当前窗口仍然可信、主题相对集中时，生成下一窗口继续工作所需的最小高保真事实源。

如果当前持续任务已经启用 `todolist.md`，则本 Skill 还必须：

> 在生成 `handoff-current.md` 的同时，根据当前窗口进展和 TodoList 规则，同步输出更新后的 `todolist.md`。

职责边界：

- `handoff-current.md` = 当前上下文快照；
- `todolist.md` = 主线推进账本；
- 本 Skill = 换窗口主流程；
- `04a-goal-todolist.md` = TodoList 的结构与同步语义来源。

handoff 可以引用和摘要 TodoList，但不要完整复制 TodoList 正文，避免两个文件互相打架。

## 工作原则

### 1. 先判断是否适合常规版

生成 handoff 前，必须先判断当前窗口是否适合直接总结。

适合常规版的条件：

- 当前窗口上下文基本完整；
- 主题集中；
- 关键结论没有严重冲突；
- 不需要追溯大量旧聊天；
- 不需要对旧窗口进行去毒纠偏。

如果不满足，应提醒用户改用中转蒸馏版。

### 2. 先识别任务是否启用了 TodoList

生成 handoff 前，必须判断当前任务是否已经启用 `todolist.md`。

- 未启用 `todolist.md`：只生成 `handoff-current.md`。
- 已启用 `todolist.md`：必须同时读取旧 `todolist.md`，并输出更新后的 `todolist.md`。

如果用户准备换窗口但忘记提供已有 `todolist.md`，应提醒：

> 当前任务似乎已经启用 TodoList，请同时提供 todolist.md；如果没有启用，则本次只生成 handoff-current.md。

### 3. 输出当前状态快照，不写聊天流水账

handoff 应是当前状态快照，不是过程复述。

必须保留：

- 最终目标；
- 当前阶段目标；
- 当前状态；
- 已确认结论；
- 用户强偏好；
- 关键规则；
- 禁止事项；
- 已知风险；
- 证据要求；
- 未决问题；
- 下一步原子任务；
- 如果已启用 TodoList，则保留配套 `todolist.md` 的摘要和使用规则。

必须删除或压缩：

- 已解决的问题；
- 已完成且无后续影响的阶段细节；
- 重复讨论；
- 情绪性表达；
- 无关背景；
- 与当前继续工作无关的工具探索。

### 4. 防止 handoff 膨胀

handoff 不是越长越好。

压缩的是流水账，不是关键事实。

如果已启用 TodoList，handoff 不要完整内嵌 TodoList，只写：

- 配套文件名；
- 当前主线；
- 当前阶段；
- 下一步原子任务；
- 冲突处理规则；
- 新窗口必须同时读取两个文件。

### 5. 区分事实、判断和待确认

handoff 中必须区分：

- 已确认事实；
- 用户明确偏好；
- 当前判断；
- 待确认事项；
- 冲突或不确定点。

不能把推测写成事实。

如果某个项目事实需要执行端、仓库、日志或截图验证，必须写成：

> 待执行端提供证据确认。

### 6. 同步更新 TodoList 时只做滚动更新

如果输入了旧 `todolist.md`，只允许基于当前窗口新增证据做滚动更新。

可以做：

- 根据用户确认或证据，将 Todo 项状态更新为 `[已完成]`、`[进行中]`、`[阻塞]`、`[暂停]` 或 `[废弃]`；
- 从当前 P0 或当前阶段目标中拆出新的“下一步原子任务”；
- 把用户明确新增的方向加入大方向 TodoList；
- 把 AI 推测但未确认的新方向放入“待确认 / 候选项”；
- 把已废弃但影响后续避坑的路线移入“已完成 / 暂停 / 废弃”。

禁止做：

- 重新发散出一份新计划覆盖旧 TodoList；
- 把 doctor、README、SKILL_INDEX、模板、脚本等支撑设施维护提升为主线目标；
- 把一次性命令或文件修改清单塞进大方向 TodoList；
- 删除旧项而不说明原因；
- 把待确认项自动升级成 P0；
- 为了文档漂亮隐藏冲突。

### 7. 多轮滚动更新时防止越滚越大

如果用户提供旧 handoff 和旧 TodoList，并要求生成新文件：

1. 以旧 handoff 和旧 TodoList 为当前基线；
2. 合并当前窗口新增有效事实；
3. 删除或归档已完成、已过期、已解决或无后续影响的内容；
4. 不把旧 handoff 原文拼接进新 handoff；
5. 不把旧 TodoList 变成流水账；
6. 新文件应该是新的当前状态快照和新的主线推进账本。

## 推荐文件名

默认输出：

```text
handoff-current.md
```

如果当前任务启用了 TodoList，同时输出：

```text
todolist.md
```

## 推荐输出结构

### 0. 适用性判断

说明当前 handoff 基于什么生成，并判断是否适合常规版。

### 1. 当前主题

用一两句话说明本 handoff 要继续的主题。

### 2. 最终目标

说明用户最终想达成什么，而不是只写当前小任务。

### 3. 当前阶段目标

说明当前阶段只解决什么，不解决什么。

### 4. 当前状态

说明已经推进到哪里。

### 5. 已确认结论

列出仍然有效的关键结论。

### 6. 用户偏好与长期规则

只写与当前主题有关的偏好。

### 7. 禁止事项

写清后续不能做什么。

### 8. 关键文件 / 路径 / 名称

仅当当前主题涉及项目、仓库、文件或命令时保留。

### 9. 证据与复核要求

说明哪些结论已有证据，哪些还需要证据。

### 10. 冲突 / 待确认

记录不能确定、互相冲突或需要下个窗口先确认的内容。

### 11. 配套 TodoList

仅当当前任务启用了 `todolist.md` 时保留。

必须说明：

- 配套文件名：`todolist.md`；
- 当前主线；
- 当前阶段；
- 下一步原子任务；
- 新窗口必须同时读取 `handoff-current.md` 和 `todolist.md`；
- 如果两者冲突，先指出冲突，不要自行合并。

### 12. 下一步原子任务

写清下一窗口应该从哪里继续。

要求：

- 具体；
- 可执行；
- 不要开太多分支；
- 不要把已经完成的事情再做一遍；
- 如果有 TodoList，必须从当前 P0 或当前阶段目标中拆出。

### 13. 新窗口启动提示

最后提供一段可直接复制到新窗口的启动提示。

如果有 `todolist.md`，启动提示必须要求新窗口同时读取两个文件。

## 输出风格

- 中文；
- 结构清晰；
- 结论先行；
- 少写废话；
- 保留关键细节；
- 不写流水账；
- 明确不确定点；
- 优先输出 `.md` 文件；
- 如果同时输出 handoff 和 TodoList，优先生成 `.zip`。

## 禁止事项

- 禁止把 handoff 写成聊天摘要。
- 禁止把 handoff 写成流水账。
- 禁止只给轻量方向总结。
- 禁止遗漏关键规则、边界、证据要求和待确认项。
- 禁止多轮 handoff 直接拼接，导致越滚越大。
- 禁止把猜测写成事实。
- 禁止假装已经读过未提供的项目、仓库、日志或文件。
- 禁止在 handoff 中引入用户没有确认的新工具路线。
- 禁止把“读取 handoff 的启动提示”和“生成 handoff 的完整规则”混在一起。
- 禁止在复杂 handoff 中使用容易复制损坏的嵌套 Markdown 大代码块。
- 禁止在已启用 TodoList 的持续任务中，只输出 handoff 而忽略 `todolist.md`。
- 禁止把 TodoList 完整复制进 handoff，造成两个事实源互相冲突。
