# ChatGPT 专属 Skill：常规新窗口交接与 Handoff

## 触发场景

当用户明确提出以下需求时，启用本 Skill：

- 生成 handoff / 新窗口交接文档；
- 准备普通换窗口；
- 整理当前窗口继续工作的上下文；
- 基于当前窗口生成 `handoff-current.md`；
- 基于当前窗口生成 `handoff-evidence.md`；
- 基于已读取的旧 `handoff-current.md` / `handoff-evidence.md` / `todolist.md` 生成滚动更新版交接包；
- 当前持续任务已经启用 `todolist.md`，需要在换窗口时同步 checkbox 主线清单。

注意：触发意图不等于一定适合常规版。生成前必须先做适用性判断。

## 不触发场景

以下情况不要启用本 Skill：

- 用户只是一次性问答、简单解释或临时帮助；
- 用户只是评审、修改、重构、调试 handoff Skill / Template / Prompt；
- 用户在新工作窗口上传或粘贴已有 handoff，要求读取、确认或继续工作；
- 用户明确要求从完整旧聊天记录、旧窗口导出文件中做去毒蒸馏；
- 当前窗口已经很长、很乱、跨主题严重，或用户认为当前窗口已降智、复读、顺从、胡乱修复。

如需从旧窗口完整材料中剔除污染并生成新 handoff，应改用 `skills/04b-handoff-distillation.md`。

如果用户上传的是已经整理过的旧 handoff / handoff 包，并要求在当前可靠窗口中滚动更新，不属于中转蒸馏；可以使用本 Skill，但必须以旧 handoff 为基线，不要重新摘要当前聊天过程。

## 核心定位

常规 handoff 只适用于：当前窗口仍可靠、主题相对集中、关键结论没有明显冲突。

它的目标是生成下一窗口继续工作所需的当前状态快照与完整可迁移证据：

- `handoff-current.md`：当前上下文快照；
- `handoff-evidence.md`：完整可迁移证据归档、原始材料定位、证据缺口与复核要求；
- `todolist.md`：仅当持续任务已经启用 TodoList，且用户提供了当前文件或当前窗口刚按 `skills/04a-goal-todolist.md` 得到明确确认时，同步极简 checkbox 主线清单。

`handoff-current.md` 是当前状态快照，不是当前窗口聊天摘要。若当前窗口读取过旧 handoff / handoff 包，新文件应是“旧可信 handoff 的滚动更新版”，而不是“当前窗口重新总结版”。

`handoff-evidence.md` 不应该只是证据摘要。凡是当前窗口中可复制、可摘录、可迁移的关键文本证据，都应尽量完整写入，例如用户确认原话、关键 Prompt、关键日志、报错、diff、命令输出、文件片段、读取范围和结论依据。

如果存在旧 `handoff-evidence.md`，新 `handoff-evidence.md` 必须做证据基线融合：继承仍支撑当前结论的旧证据；补入当前窗口新增确认；把被用户否定或已失效的旧证据移入“不继承 / 已废弃证据”；对旧证据与新结论冲突处显式标注，不自行调和。证据融合不是把旧 evidence 原文与当前聊天记录简单拼接。

`handoff-evidence.md` 的“完整”不等于保存聊天噪音。它应保存支撑当前结论和下一步判断的关键证据；助手跑偏、被用户否定、已废弃的中间判断，只能作为“不继承 / 已废弃路径”短证据记录，不得进入当前状态、已确认结论或 TodoList。

`handoff-evidence.md` 也不是原始附件替代品。图片、截图、zip、源码包、视频画面、本地路径指向文件等无法完整写入 Markdown 的材料，不能假设会自动迁移。遇到这类关键材料时，应优先在生成前或最终回答中提示用户补充；若用户明确要求继续生成，必须在 `handoff-evidence.md` 中标出“未完整携带 / 待补证”，但不要为此额外生成独立补证清单文件。

本 Skill 不重新定义 TodoList 语义。涉及 TodoList 时，必须引用并遵守 `skills/04a-goal-todolist.md`。当前 TodoList 语义为：`todolist.md` 只是一份极简 checkbox 主线清单，不扩展成项目管理文档。

## 工作原则

### 1. 先做 Handoff 生成审核卡

生成文件前，先输出简短审核卡，停止等待用户确认，除非用户明确要求直接生成。

审核卡默认结构：

```text
Handoff 生成审核卡：
- 当前任务类型：生成 / 读取 / 评审 / 修改 / 不适用
- 是否适合常规 handoff：是 / 否
- 是否需要中转蒸馏：是 / 否
- 是否存在旧 handoff / handoff 包作为基线：是 / 否 / 不确定
- 基线文件是否已读取：是 / 否 / 不适用
- 是否检测到配套 TodoList：是 / 否 / 不确定
- 当前窗口可迁移证据是否足够生成 handoff-evidence.md：是 / 否 / 不确定
- 下一窗口继续工作是否缺关键外部材料：是 / 否 / 不确定
- 需用户补充的外部材料：无 / 列表
- 建议动作：直接生成 / 以旧 handoff 为基线滚动生成 / 直接生成但最终回答提示补充外部材料 / 先补充材料再生成 / 改用中转蒸馏 / 本次只做评审
```

如果判断为读取、评审、修改或不适用，不要继续生成 handoff 文件。

如果用户明确要求直接生成，也必须在内部先做以上判断，并在输出中体现关键适用性结论。

审核卡中的两个证据判断必须分开：

- `当前窗口可迁移证据是否足够生成 handoff-evidence.md`：判断当前窗口里已经可见、可复制、可摘录的文本证据是否足够写入证据归档；
- `下一窗口继续工作是否缺关键外部材料`：判断下一窗口要继续诊断、复核或执行时，是否仍需要重新上传截图、图片、zip、源码包、完整日志、diff、视频画面、本地文件或当前 Prompt 等外部材料。

允许同时出现“可生成 handoff-evidence.md = 是”与“下一窗口缺关键外部材料 = 是”。这表示当前窗口可迁移证据足够生成交接包，但新窗口继续工作仍需用户补充外部原件。不要写成互相矛盾的表达。

### 2. 先识别基线，再生成内容

生成前必须判断本次属于哪种模式：

```text
A. 无旧 handoff：基于当前可靠窗口生成当前状态快照。
B. 有旧 handoff / handoff 包：以最新可信 `handoff-current.md` 为主基线，滚动融合当前窗口新增确认。
C. 有完整旧聊天记录或旧窗口污染严重：不使用常规版，改用中转蒸馏。
```

如果存在旧 `handoff-current.md`、`handoff-evidence.md`、`todolist.md` 或 handoff zip 包，并且当前窗口已读取或识别它们：

1. `handoff-current.md` 必须以旧 `handoff-current.md` 的仍有效内容为主体；
2. 当前窗口只合并用户明确确认的新结论、被明确废弃的旧结论、TodoList 状态变化、新增证据缺口和下一步变化；
3. 不要把当前窗口的争论、跑偏、反复解释、错误尝试写成主线；
4. 如果旧 handoff 与当前窗口确认结论冲突，必须在“冲突 / 待确认”中列出，不要自行调和；
5. 如果旧 handoff 原件没有被完整读取，只能说明“旧 handoff 未完整携带 / 需补充”，不能假装已经融合。

### 3. 输出当前状态快照，不写聊天流水账

`handoff-current.md` 应保留下一窗口继续工作必须知道的内容：

- 当前主题；
- 最终目标；
- 当前阶段目标；
- 当前状态；
- 已确认结论；
- 与当前主题相关的用户偏好与长期规则；
- 禁止事项 / 废弃路径；
- 关键文件、路径、命令或名称；
- 证据与复核要求；
- 冲突 / 待确认；
- 下一步建议；
- 如已启用 TodoList，只摘要 checkbox 主线清单和读取要求，不写复杂优先级、阶段拆解或项目管理结构。

应删除或压缩：

- 已解决且无后续影响的过程；
- 重复讨论；
- 情绪性表达；
- 与当前继续工作无关的工具探索；
- 助手被用户明确否定的错误判断；
- 用户已经要求不要继承的旧窗口错误尝试；
- 已完整归档到 `handoff-evidence.md` 的证据正文。

证据正文放入 `handoff-evidence.md`，不要塞满 `handoff-current.md`。

### 4. 默认同时生成完整证据归档，但必须做证据基线融合与去噪

生成 `handoff-current.md` 时，默认同时生成 `handoff-evidence.md`。

`handoff-evidence.md` 必须优先保存完整可迁移证据，而不是只写摘要，包括：

- 用户明确确认的原话或完整结论；
- 已粘贴的关键日志、报错、diff、命令输出；
- 已读取文件、Prompt、Skill、模板的关键原文、完整片段和读取范围；
- 已讨论并稳定识别的图片 / 截图信息；
- 原始材料定位卡；
- 未完整携带的外部证据、缺口和复核方式。

但以下内容不要作为正文展开：

- 已被用户否定的助手误判；
- 为了纠错而出现的来回争论；
- 已放弃方案的完整推理过程；
- 对下一窗口没有复核价值的情绪性或过程性内容。

这些内容如影响后续判断，只能简短记录到“禁止继承 / 已废弃路径 / 不需要携带的材料”。

如果关键证据在当前窗口不可见、不可摘录或无法完整迁移，应在生成前或最终回答中直接提示用户补充。不要通过新增独立补证文件来替代这个提示。

### 5. 自动识别 TodoList，但不强行创建

生成 handoff 前，判断当前任务是否已经启用 `todolist.md`。

判定依据包括：

- 用户提供了 `todolist.md`；
- 当前窗口或旧 handoff 明确声明存在配套 TodoList；
- 当前上下文持续引用 `todolist.md`、主线 TodoList、大方向 TodoList、checkbox 清单等；
- 当前窗口刚刚按 `skills/04a-goal-todolist.md` 输出候选清单，并得到用户明确确认；
- 用户明确要求按 TodoList 延续。

处理规则：

- 未检测到 TodoList：只生成 `handoff-current.md` 和 `handoff-evidence.md`；
- 检测到 TodoList 且用户提供了文件：按 `04a-goal-todolist.md` 同步输出更新后的 `todolist.md`；
- 检测到 TodoList 且当前窗口刚刚由用户确认正式 checkbox 清单：可直接输出 `todolist.md`，并在 `handoff-evidence.md` 中记录确认来源；
- 检测到 TodoList 但用户未提供文件、当前窗口也没有确认过正式清单：完整交接应先提醒补充；若用户明确只要 handoff，则可继续，但必须记录 TodoList 未同步及影响；
- 当前请求只是评审或修改 handoff Skill / Template / Prompt 时，不进入 TodoList 同步流程。

TodoList 同步只允许：

- 根据用户明确确认或可靠证据，把 `[ ]` 改成 `[x]`；
- 根据用户明确新增的主线事项，新增一条 checkbox；
- 根据用户明确废弃的主线事项，删除该条，或保留并在最终回答中说明。

禁止在 handoff 同步时把极简 TodoList 改写成目标、阶段、优先级、验收矩阵、文件清单或命令清单。

### 6. 区分事实、判断和待确认

handoff 中必须区分：

- 已确认事实；
- 用户明确偏好；
- 当前判断；
- 待确认事项；
- 冲突或不确定点。

项目事实需要仓库、执行端、日志、截图或测试结果验证时，必须标为待复核，不能写成已确认事实。

助手提出但用户未确认的分析，只能写成“当前判断”或“不继承项”，不能写成已确认结论。

### 7. 多轮滚动时防止越滚越大

如果用户提供旧 handoff 或旧 TodoList，并要求生成新文件：

1. 以最新可信文件为基线；
2. 合并当前窗口新增的有效事实；
3. 删除或归档已完成、已过期、已解决或无后续影响的内容；
4. 删除或压缩当前窗口中被用户纠正的跑偏过程；
5. 不拼接旧文件原文；
6. 不拼接当前聊天流水；
7. 新文件必须是当前状态快照、完整可迁移证据归档和极简主线清单，而不是历史合集。

滚动更新的判断顺序：

```text
旧 handoff 仍有效内容
→ 当前窗口用户明确确认的新结论
→ 当前窗口明确废弃 / 不继承内容
→ 当前窗口新增证据缺口
→ 下一步建议
```

不要反过来从当前窗口聊天记录重写旧 handoff。

## 推荐输出文件

默认输出：

```text
handoff-current.md
handoff-evidence.md
```

如果当前任务已经启用 TodoList，且用户提供了 `todolist.md` 或当前窗口刚确认正式 TodoList，同时输出：

```text
todolist.md
```

如果同时输出两个或以上文件，优先生成 `.zip`。

如果缺少关键图片、附件、zip、日志、diff、Prompt 或本地文件，优先在聊天最终回答中列出需要用户补充的外部材料；不要默认额外生成独立补证文件。

## 推荐输出结构

### handoff-current.md

```markdown
# Handoff Current

## 0. 适用性判断
## 1. 基线来源与融合方式
## 2. 当前主题
## 3. 最终目标
## 4. 当前阶段目标
## 5. 当前状态
## 6. 已确认结论
## 7. 用户偏好与长期规则
## 8. 禁止事项 / 废弃路径
## 9. 关键文件 / 路径 / 名称
## 10. 证据材料包
## 11. 证据与复核要求
## 12. 冲突 / 待确认
## 13. 配套 TodoList（仅在已启用时保留）
## 14. 下一步建议
```

`基线来源与融合方式` 章节必须说明：本次是否有旧 handoff 作为基线；如果有，旧 handoff 是否完整读取；当前窗口只融合了哪些新增确认。

`配套 TodoList` 章节只摘要 checkbox 主线清单，不扩写成复杂结构。

### handoff-evidence.md

优先使用 `templates/handoff-evidence-pack.md` 的结构。至少包含：

1. 本文件用途；
2. 证据融合模式与基线审计；
3. 完整可迁移证据；
4. 原始材料定位；
5. 未完整携带 / 需要外部补充的证据；
6. 证据缺口 / 待复核；
7. 不需要携带 / 禁止继承的材料；
8. 新窗口读取建议。

## 输出风格

- 中文；
- 结论先行；
- 高密度；
- 少背景；
- 不写流水账；
- 明确不确定点；
- 下一步建议保持短小；
- 证据正文放入 `handoff-evidence.md`；
- 优先输出 `.md` 文件；多个文件优先 `.zip`。

## 禁止事项

- 禁止把 handoff 写成聊天摘要或历史合集。
- 禁止把当前窗口过程当成旧 handoff 的替代品。
- 禁止读取过旧 handoff 却不以它为基线滚动更新。
- 禁止把旧 handoff 原文拼接进新 handoff。
- 禁止把当前聊天流水拼接进新 handoff。
- 禁止省略必要的 `handoff-evidence.md`。
- 禁止把 `handoff-evidence.md` 写成只有摘要、没有关键证据正文的索引。
- 禁止把助手跑偏、被用户否定、已废弃的中间判断写入当前状态、已确认结论或 TodoList。
- 禁止把不可迁移、不可复核的材料写成已完整携带证据。
- 禁止为补证清单默认新增独立补证清单文件。
- 禁止强行创建不存在的 TodoList。
- 禁止检测到已有 TodoList 却忽略同步或缺口说明。
- 禁止在本 Skill 内重新定义 TodoList 语义。
- 禁止把极简 TodoList 膨胀成复杂项目管理结构。
- 禁止为了整洁隐藏冲突或证据缺口。
