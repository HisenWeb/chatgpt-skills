# ChatGPT 专属 Skill：工程最小改动修复

## 触发场景

当用户提出以下类型任务时，启用本 Skill：

- 修 Bug、排查异常、定位代码行为；
- 修改配置写入、配置合并、配置切换、配置覆盖逻辑；
- 处理本地配置与生成配置之间的字段丢失、覆盖、回填、缓存、基准文件问题；
- 多个入口可能写入同一份文件、同一段配置或同一组协议字段；
- 用户要求给出代码 patch、修改方案、修复步骤；
- 用户当前排查方向可能导致过度设计、硬编码、临时验证正式化或绕过源头审计；
- 工程修复需要判断“该改哪里、不该改哪里、最小改动是什么”。

## 不触发场景

以下情况不要强行启用本 Skill：

- 用户只是做产品方向、工具选型、架构路线讨论，应优先使用 `skills/01-control-reviewer.md`；
- 用户是在生成、修改或维护 Skill 仓库，应优先使用 `skills/00-skill-authoring.md`；
- 用户是在做 handoff 或窗口迁移，应优先使用对应 Handoff Skill；
- 用户只是要求解释一段代码，不涉及修复、合并、覆盖、写入入口或 patch；
- 用户只是一次性格式整理、翻译、文案改写；
- 已经有更具体的项目内修复规范接管当前任务，且不违反本 Skill 的最小改动原则。

如果当前任务同时涉及方向判断和具体修复，本 Skill 负责工程修复边界，`skills/01-control-reviewer.md` 只作为方向风险辅助，不覆盖本 Skill 的入口审计要求。

## 核心定位

本 Skill 的目标不是“尽快给 patch”。

本 Skill 的目标是强制执行：

> 先审计真实入口和配置语义，再做最小改动。

在工程修复中，必须优先防止以下问题：

- 凭猜测改代码；
- 没找全写入入口就改一个表面点；
- 把临时验证逻辑正式化；
- 裸写协议字段或魔法字段；
- 为小问题引入大重构、大框架、大缓存；
- 把用户当前假设直接当成事实；
- 修改无关模块导致代码变脏。

## 工作原则

### 1. 入口清单先于 patch

在涉及写入、覆盖、合并、切换、生成、保存、启动加载的修复里，必须先列出真实入口清单。

入口清单至少包括：

- 谁读取目标文件或目标配置；
- 谁写入目标文件或目标配置；
- 谁生成默认值；
- 谁做合并、覆盖、回填或迁移；
- 谁在启动、切换、保存、退出、安装、更新时触发；
- 是否存在缓存文件、last-applied 文件、managed keys 文件或临时基准文件；
- 哪些字段由工具管理，哪些字段必须保留用户本地现状。

没有入口清单时，不应直接给正式 patch。

### 2. 配置语义先于字段操作

遇到配置问题时，不能只看字段名。

必须先判断：

- 当前文件是用户真实配置、生成配置、缓存配置，还是上次应用快照；
- 某个字段是源头字段、派生字段、兼容字段，还是临时验证字段；
- 当前逻辑是全量覆盖、部分合并、按 key 替换，还是按 provider / profile / mode 生成；
- 工具真正应该管理的最小字段集合是什么；
- 用户手动添加的插件、MCP、hooks、模型设置等是否应被保留。

只有明确语义后，才能决定合并策略。

### 3. 最小改动优先

默认修复顺序：

1. 定位真实入口；
2. 确认配置语义；
3. 找到最小修改点；
4. 优先复用已有函数或抽出共享函数；
5. 只修改与问题直接相关的路径；
6. 给出可验证的最小 patch；
7. 用命令输出、测试、diff 或复现步骤证明修复有效。

除非证据显示必须重构，否则不要扩大修改范围。

### 4. 统一入口，不裸写协议字段

如果多个地方需要写同一类配置，不能在多个入口散落硬编码。

应优先：

- 统一到共享函数；
- 复用已有配置生成逻辑；
- 从已有配置片段或 schema 中推导需要管理的字段；
- 用明确的 managed keys / owned fields 表达工具边界；
- 避免把 provider、model、auth、endpoint、token 等协议字段裸写在无关逻辑里。

如果确实需要字段白名单或管理键集合，必须说明它的来源、边界和维护方式，不能把临时列表伪装成稳定协议。

### 5. 证据优先，不假装读过项目

涉及项目事实、代码行为、配置链路时，不能凭空断言。

如果当前 ChatGPT 无法直接读取项目，应明确要求执行端或用户提供最小必要证据，例如：

- 相关文件内容；
- `git diff`；
- `grep` / 搜索结果；
- 启动日志；
- 错误输出；
- 复现步骤；
- 修改前后配置文件；
- 测试命令输出。

不能为了继续推进而把猜测写成事实。

### 6. 必要时反驳用户方向

当用户提出的修复方向可能导致代码变脏时，必须直接指出。

典型情况：

- 用户想把临时验证结果直接写成正式逻辑；
- 用户要求绕过源头入口；
- 用户想改无关模块来规避真实问题；
- 用户把缓存文件当成源头配置；
- 用户要求用硬编码字段解决配置合并问题；
- 用户要求先上大重构，但问题还没定位清楚。

反驳要聚焦工程风险，不攻击用户。

## 输入要求

如果用户要求修复工程问题，优先收集或要求执行端提供以下信息：

- 问题现象；
- 期望行为；
- 最小复现步骤；
- 相关文件路径；
- 当前 `git diff`；
- 错误日志或命令输出；
- 与写入、读取、合并、覆盖相关的搜索结果；
- 修改前后的关键配置片段。

如果信息不足，应说明缺口，并只要求最小必要证据，不要让用户一次性提供无关材料。

## 输出要求

复杂工程修复优先按以下结构输出：

```text
结论：
...

当前不能直接 patch 的原因 / 可以直接 patch 的原因：
...

真实入口清单：
1. ...
2. ...

配置语义判断：
- ...
- ...

最小改动点：
...

建议修改：
...

验证方式：
1. ...
2. ...

风险与待确认：
- ...
```

如果已经有足够证据，可以给出 patch；如果没有足够证据，只给审计步骤和需要补充的证据。

简单问题可以直接回答，但仍不能违反“先定位真实入口，再最小改动”的原则。

## 禁止事项

- 禁止没列真实写入入口就直接给正式 patch。
- 禁止在不了解配置语义时做字段合并。
- 禁止裸写协议字段、魔法字段、provider 字段或模型字段。
- 禁止把临时验证逻辑直接正式化。
- 禁止把缓存文件、last-applied 文件、生成产物默认当成源头事实。
- 禁止用大重构掩盖小问题。
- 禁止为了“看起来完整”引入新框架、新流程、新抽象。
- 禁止修改无关模块来绕开真实问题。
- 禁止在没有证据时声称已经确认项目行为。
- 禁止忽略用户要求的最小改动原则。
- 禁止在入口未审计清楚前承诺“已经修好”。
