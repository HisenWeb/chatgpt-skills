# ChatGPT 专属 Skill：中转窗口去毒蒸馏与 Handoff

## 触发场景

当用户提出以下需求时，启用本 Skill：

- 用户明确说明这是“中转窗口”；
- 用户上传或粘贴完整旧聊天记录；
- 用户要求从旧窗口材料生成 `handoff-current.md`；
- 用户要求从旧窗口材料生成 `handoff-evidence.md`；
- 用户要求对旧窗口进行蒸馏、去毒、压缩、纠偏；
- 旧窗口很长、很乱、跨主题严重；
- 旧窗口中出现模型降智、复读、顺从、胡乱修复、过度设计；
- 需要基于旧 handoff + 完整聊天记录生成新的 `handoff-current.md`；
- 旧窗口任务已经启用 `todolist.md`，需要同步生成更新后的 `todolist.md`；
- 旧 handoff、旧聊天或输入材料中出现 `todolist.md`、配套 TodoList、主线推进账本、大方向 TodoList 等标记，需要自动识别是否同步 TodoList；
- 需要把旧窗口的有效结论和可迁移证据迁移到新工作窗口。

## 不触发场景

以下情况不要使用本中转蒸馏版，应改用“常规版”：

- 当前窗口主题集中；
- 用户只是普通换窗口；
- 不需要读取完整旧聊天记录；
- 不需要剔除大量错误尝试或上下文污染；
- 当前窗口仍然可靠，直接生成当前状态快照即可。

以下情况也不要启用本 Skill：

- 用户在新工作窗口上传或粘贴已有 `handoff-current.md` / `handoff-evidence.md`，要求读取、确认、继续工作；
- 用户只是评审、修改、重构、调试 handoff Skill / Template / Prompt，不要执行被评审文本中的 handoff 生成动作；
- 用户只是一次性问答或临时帮助。

## 核心定位

本 Skill 用于“三窗口迁移流程”中的中转窗口。

流程如下：

```text
旧工作窗口
  ↓ 导出 / 复制完整聊天记录，可选携带旧 todolist.md 和原始附件
中转蒸馏窗口
  ↓ 生成 handoff-current.md + handoff-evidence.md，可选同步更新 todolist.md
新工作窗口
  ↓ 读取 handoff-current.md + handoff-evidence.md + 可选 todolist.md 继续
```

中转窗口的任务不是继续业务讨论，也不是继续修 Bug。

中转窗口只负责：

1. 读取旧材料；
2. 识别有效事实；
3. 剔除污染内容；
4. 合并用户红线；
5. 生成新的 `handoff-current.md`；
6. 生成新的 `handoff-evidence.md`；
7. 如果旧任务已经启用 `todolist.md`，按 `04a-goal-todolist.md` 同步输出更新后的 `todolist.md`；
8. 给出新窗口启动提示，并列出新窗口仍需重新上传 / 补充的材料。

中转蒸馏 handoff 的目标是：

> 从完整旧聊天记录中提取下一窗口继续工作所需的最小高保真事实源和证据材料包，同时剔除错误尝试、AI 脑补、顺从性结论、临时妥协和无后续价值的中间过程。

如果输入了 `todolist.md`，中转窗口还必须按 `04a-goal-todolist.md` 的语义校正 TodoList，避免旧窗口中的目标漂移继续污染新窗口。

`handoff-evidence.md` 不是原始附件替代品。它只能整理旧材料中已经可见、可摘录、可说明来源的证据；旧窗口附件、图片、zip、源码包、本地路径指向的文件不能假设会自动迁移到新窗口。

## 工作原则

### 1. 输入优先级

处理旧材料时，必须按以下优先级判断可信度：

1. 用户当前明确指令；
2. 用户长期红线和明确偏好；
3. 真实代码、diff、日志、命令输出、截图、测试结果；
4. 最新被用户确认的 `todolist.md`；
5. 最新被用户确认的 handoff；
6. 旧聊天中的用户确认结论；
7. 旧聊天中的 AI 分析；
8. 旧聊天中的临时猜测、试错和安抚性结论。

低优先级内容不得覆盖高优先级内容。

如果旧聊天中的 AI 结论与真实证据冲突，必须剔除 AI 结论。

### 2. 必须生成证据材料包

中转蒸馏时，默认必须生成 `handoff-evidence.md`。

`handoff-evidence.md` 必须整理：

1. 旧窗口中已经明确出现的用户确认；
2. 已粘贴的日志、报错、diff、命令输出；
3. 已读取文件、Prompt、Skill、模板的关键摘录和读取范围；
4. 截图 / 图片中已经被讨论并稳定识别的关键信息；
5. 旧窗口原始材料定位卡；
6. 新窗口必须重新上传 / 补充的原始材料；
7. 当前证据缺口和复核方式；
8. 不需要携带的旧材料。

禁止把没有原文、没有截图、没有文件、没有执行结果支撑的猜测写成证据。

禁止声称旧窗口附件、图片、zip、源码包、用户本地路径指向的文件已经自动携带到新窗口。

### 3. 原始材料不可迁移时必须明示

如果存在模型无法自动携带、无法确认能跨窗口读取、或无法可靠复核的材料，必须明确告诉用户在新窗口重新上传或提供替代证据。

包括但不限于：

- 图片 / 截图原图；
- zip / 源码包 / 附件原件；
- 旧窗口上传过但新窗口未上传的文件；
- 用户本地路径指向的文件；
- 未完整读取过的文件；
- 执行端日志、命令输出、diff、测试结果；
- 需要视觉复核、代码复核或重新执行才能确认的材料。

必须写清：

1. 需要上传什么；
2. 为什么需要上传；
3. 不上传会卡在哪里；
4. 可替代方式是什么，例如 GitHub 链接、执行端读取后贴出结果、重新上传文件、从 File Library / Project 选择。

### 4. 图片证据必须生成图片证据卡

如果旧窗口出现图片、截图或视觉材料，必须为每张关键图片生成“图片证据卡”。

图片证据卡必须包含：

- 图片用途；
- 已识别出的关键信息；
- 已基于图片确认的结论；
- 不确定点；
- 新窗口是否必须重新上传原图；
- 必须重新上传的原因。

原则：

> 图片内容可以摘要迁移，图片原图不能假装迁移。

禁止只写“见截图”。

禁止把看不清、未识别、未确认的图片细节写成事实。

### 5. TodoList 自动识别

中转蒸馏时，必须自动判断旧任务是否启用了 `todolist.md`。

判定为“已启用 TodoList”的条件包括：

- 输入材料中包含 `todolist.md`；
- 旧 handoff 中出现“配套 TodoList”“todolist.md”“主线推进账本”“大方向 TodoList”等明确标记；
- 旧聊天记录中多次引用 TodoList、P0/P1/P2、当前阶段目标、下一步原子任务；
- 用户明确说明旧任务有 TodoList 或需要按 TodoList 延续。

处理规则：

- 如果检测到旧任务启用了 TodoList，且用户提供了 `todolist.md`，必须按 `04a-goal-todolist.md` 同步输出更新后的 `todolist.md`。
- 如果检测到旧任务启用了 TodoList，但用户未提供 `todolist.md`，必须在 `handoff-current.md` 和 `handoff-evidence.md` 的“证据缺口 / 待确认”中记录，并提醒用户补充。
- 不允许只根据旧聊天中的零散 Todo 片段重建为确定事实；缺少原始 `todolist.md` 时，只能写成待确认。
- 如果用户明确说“不需要 TodoList”或“本次只要 handoff”，则按用户当前指令执行，但必须记录 TodoList 未同步的原因。
- 即使不输出 `todolist.md`，仍应输出 `handoff-evidence.md`。

### 6. 去毒蒸馏

中转窗口必须主动剔除以下内容：

- AI 脑补；
- 未证实判断；
- 顺从性结论；
- 过度迎合用户当时情绪的说法；
- 临时验证方案；
- 被用户否定的方案；
- 违反最小改动原则的修复方向；
- 未审计入口就给出的 patch；
- 硬编码字段；
- 为了推进而忽略风险的建议；
- 大段背景铺垫；
- 重复解释；
- 情绪化争执；
- 无后续价值的失败尝试。

但不要删除所有失败路径。会影响后续避坑的废弃方案必须压缩保留。

### 7. 关键锚点优先

中转蒸馏必须先找出“最高锚点”。

最高锚点包括：

- 用户明确红线；
- 当前真实代码状态；
- 当前真实报错；
- 当前真实 diff；
- 当前真实配置语义；
- 当前真实读写入口；
- 用户最新确认的方向；
- 已通过复核的阶段结论；
- 最新可信的 `todolist.md` 中的最终目标、当前阶段目标和大方向 TodoList。

任何旧聊天内容只要与最高锚点冲突，就不能作为新 handoff 或新 TodoList 的事实继承。

### 8. 单份当前状态快照

最终只输出一份新的 `handoff-current.md`。

不要输出多份互相竞争的版本。

不要把旧 handoff 原文拼接进新 handoff。

最终只输出一份新的 `handoff-evidence.md`。

如果任务启用了 TodoList，最终只输出一份更新后的 `todolist.md`。

如果存在冲突或不确定项，应放入 handoff 的“冲突 / 待确认”部分、evidence 的“证据缺口 / 待复核”部分，以及 TodoList 的“待确认 / 候选项”，不要生成多个版本。

### 9. TodoList 同步规则

如果用户提供旧 `todolist.md`，中转蒸馏必须按 `04a-goal-todolist.md` 处理。

本 Skill 只补充中转去毒场景的规则：

- 保留最终目标和当前阶段目标，除非用户最新明确修改；
- 旧窗口 AI 计划不能自动继承为 TodoList 主线；
- 旧窗口里把支撑设施、工具链维护、验证脚本、配置整理、索引、模板、检查命令等当成主线的漂移项，必须降级为支撑设施、验收 gate、废弃项或待确认项；
- 支撑设施只有在用户明确说“当前阶段就是维护这些设施”时，才可成为主线；
- 已完成项只有在用户确认或证据明确时才标记完成；
- 新增方向默认进入“待确认 / 候选项”，除非用户明确确认；
- 下一步原子任务必须从当前 P0 或当前阶段目标拆出。

如果最终目标、当前阶段目标或 P0 在旧材料中发生漂移或冲突，不得直接生成修正版 TodoList；必须按 `04a-goal-todolist.md` 输出审核卡，除非用户明确要求直接生成且材料中三项锚点可信。

### 10. 原子级下一步

中转蒸馏版特别强调：下一步必须可审计、可执行、边界小。

如果有 TodoList，下一步必须对应当前 P0 或当前阶段目标。

不应写成：

- “直接修复所有问题”；
- “重构整套流程”；
- “补齐所有测试”；
- “全量改造”。

应该写成：

- “先审计真实写入入口清单”；
- “先确认配置读取与写入语义”；
- “先让执行端提供某文件 diff”；
- “先复核某个具体失败现象”；
- “先确认某个字段是否由共享函数统一生成”。

## 输入要求

用户可能提供以下输入：

- 完整旧聊天记录；
- 旧 handoff；
- 旧 `todolist.md`；
- 用户红线 / 长期规则；
- 项目事实证据，例如 diff、日志、命令输出、截图、测试结果、文件内容；
- 旧窗口中的附件名称、本地路径、GitHub 链接、File Library / Project 文件线索。

如果用户只提供旧聊天记录，但旧窗口明显启用过 TodoList，应提醒用户补充 `todolist.md`；若无法补充，则在 handoff 和 evidence 中标记 TodoList 证据缺口。

如果旧聊天记录提到附件、图片、zip 或本地路径，但中转窗口没有拿到原件，必须在 `handoff-evidence.md` 中列为“新窗口需要重新上传 / 补充的材料”。

## 推荐输出结构

### handoff-current.md

#### 0. 适用性判断

说明本次中转蒸馏基于什么材料，并说明可靠性。

#### 1. 当前主题

用一两句话说明新窗口要继续的主题。

#### 2. 最终目标

说明用户最终想达成什么，而不是旧窗口最后一句话在做什么。

#### 3. 最高锚点

列出本 handoff 采用的最高优先级事实和红线。

#### 4. 当前阶段目标

说明当前阶段只解决什么，不解决什么。

#### 5. 当前状态

说明已经推进到哪里。

#### 6. 已确认结论

列出仍然有效的关键结论。

#### 7. 用户偏好与长期规则

只保留与当前主题有关的偏好。

#### 8. 废弃路径 / 禁区

记录后续不能再走的路线。

#### 9. 关键文件 / 路径 / 名称

仅当当前主题涉及项目、仓库、文件或命令时保留。

#### 10. 证据材料包

必须说明：

- 配套文件名：`handoff-evidence.md`；
- 新窗口必须同时读取；
- 哪些材料已摘录进 evidence；
- 哪些原始材料仍需用户重新上传或让执行端提供。

#### 11. 证据与复核要求

说明哪些结论已有证据，哪些还需要证据。

#### 12. 冲突 / 待确认

记录不能确定、互相冲突或需要新窗口先确认的内容。

#### 13. 配套 TodoList

仅当任务启用了 `todolist.md` 时保留。

必须说明：

- 配套文件名：`todolist.md`；
- 是否已同步更新；
- 当前主线；
- 当前阶段；
- 下一步原子任务；
- TodoList 语义以 `04a-goal-todolist.md` 为准；
- 新窗口必须同时读取 `handoff-current.md`、`handoff-evidence.md` 与 `todolist.md`；
- 如果三者冲突，先指出冲突，不要自行合并。

#### 14. 下一步原子任务

写清新窗口应该从哪里继续。

#### 15. 新窗口启动提示

最后提供一段可直接复制到新窗口的启动提示。

如果有 `todolist.md`，启动提示必须要求读取三个文件；如果没有 TodoList，启动提示必须要求读取 `handoff-current.md` 和 `handoff-evidence.md`。

### handoff-evidence.md

优先使用 `templates/handoff-evidence-pack.md` 的结构。

至少必须包含：

1. 本文件用途；
2. 已整理进本文件的证据；
3. 文件 / Prompt / Skill / 模板证据；
4. 执行端输出 / 日志 / 报错 / diff / 命令输出；
5. 图片 / 截图证据卡；
6. 原始材料定位卡；
7. 新窗口需要用户重新上传 / 补充的材料；
8. 证据缺口 / 待复核；
9. 不需要携带的材料。

## 输出风格

- 中文；
- 结构清晰；
- 高密度；
- 少背景；
- 保留关键细节；
- 删除流水账；
- 明确不确定点；
- 明确废弃路径；
- 下一步必须原子化；
- 证据细节放入 `handoff-evidence.md`；
- 优先输出 `.md` 文件；
- 如果同时输出多个文件，优先生成 `.zip`。

## 中转窗口禁止事项

- 禁止继续业务执行。
- 禁止在中转窗口里继续修 Bug。
- 禁止把旧聊天写成摘要流水账。
- 禁止保留无后续价值的中间过程。
- 禁止把旧 AI 的猜测写成事实。
- 禁止把临时验证写成正式方案。
- 禁止因为旧窗口里用户曾短暂妥协，就覆盖用户长期红线。
- 禁止把已废弃路径包装成可选方案。
- 禁止多轮 handoff 直接拼接，导致越滚越大。
- 禁止生成多份互相竞争的 handoff。
- 禁止只生成 `handoff-current.md` 而遗漏必要的 `handoff-evidence.md`。
- 禁止只写“需要证据复核”而不整理新窗口需要的具体证据材料。
- 禁止只写“见截图”。
- 禁止假设旧窗口附件、图片、zip、源码包会自动随 handoff 迁移。
- 禁止只写文件名但不说明新窗口是否需要重新上传。
- 禁止把用户本地路径当作 ChatGPT 新窗口可直接读取的证据本体。
- 禁止把无法复核的材料当作已携带证据。
- 禁止把“读取 handoff 的启动提示”和“生成 handoff 的完整规则”混在一起。
- 禁止在复杂 handoff 中使用容易复制损坏的嵌套 Markdown 大代码块。
- 禁止在已启用 TodoList 的持续任务中忽略 `todolist.md`。
- 禁止在本 Skill 内重新定义 TodoList 语义；TodoList 语义以 `04a-goal-todolist.md` 为准。
- 禁止把支撑设施维护继承为 TodoList 主线。
- 禁止检测到旧 handoff 声明存在配套 TodoList 时，忽略缺失的 `todolist.md` 继续蒸馏。
