# ChatGPT 专属 Skill：中转窗口去毒蒸馏与 Handoff

## 触发场景

当用户明确提出以下需求时，启用本 Skill：

- 这是中转窗口；
- 用户上传或粘贴完整旧聊天记录；
- 用户要求从旧窗口材料生成新的 `handoff-current.md`；
- 用户要求生成 `handoff-evidence.md`；
- 用户要求对旧窗口进行蒸馏、去毒、压缩、纠偏；
- 旧窗口很长、很乱、跨主题严重；
- 旧窗口中出现模型降智、复读、顺从、胡乱修复、过度设计；
- 需要基于旧 handoff、完整聊天记录、旧 TodoList 合并成新窗口可用材料；
- 旧任务已经启用 `todolist.md`，需要同步生成更新后的 `todolist.md`。

## 不触发场景

以下情况不要启用本 Skill：

- 当前窗口主题集中，只是普通换窗口；
- 不需要读取完整旧聊天记录；
- 不需要剔除大量错误尝试或上下文污染；
- 用户在新工作窗口上传已有 handoff，要求读取、确认或继续工作；
- 用户只是评审、修改、重构、调试 handoff Skill / Template / Prompt；
- 用户只是一次性问答或临时帮助。

普通轻量换窗口应改用 `skills/04-handoff-regular.md`。

## 核心定位

本 Skill 用于“三窗口迁移流程”中的中转窗口：

```text
旧工作窗口
  ↓ 导出 / 复制完整聊天记录，可选携带旧 todolist.md 和原始附件
中转蒸馏窗口
  ↓ 生成 handoff-current.md + handoff-evidence.md，可选同步 todolist.md
新工作窗口
  ↓ 读取 handoff-current.md + handoff-evidence.md + 可选 todolist.md 继续
```

中转窗口不是继续业务执行的窗口。它只负责：

1. 读取旧材料；
2. 识别有效事实；
3. 剔除污染内容；
4. 合并用户红线；
5. 生成新的当前状态快照；
6. 生成完整可迁移证据归档；
7. 必要时按 `skills/04a-goal-todolist.md` 同步 TodoList；
8. 在最终回答中提示新窗口仍需重新上传或补充的外部材料。

目标是从旧窗口中提取最小高保真事实源，避免错误尝试、AI 脑补、顺从性结论、临时妥协和无后续价值的过程继续污染新窗口。

`handoff-evidence.md` 不应该只是证据摘要。凡是旧材料中可复制、可摘录、可迁移的关键文本证据，都应尽量完整写入，例如用户确认原话、关键 Prompt、关键日志、报错、diff、命令输出、文件片段、读取范围和结论依据。

`handoff-evidence.md` 也不是原始附件替代品。图片、截图、zip、源码包、视频画面、本地路径指向文件等无法完整写入 Markdown 的材料，不能假设会自动迁移。遇到这类关键材料时，应优先在生成前或最终回答中提示用户补充；若用户明确要求继续生成，必须在 `handoff-evidence.md` 中标出“未完整携带 / 待补证”，但不要为此额外生成独立补证清单文件。

## 工作原则

### 1. 输入优先级

处理旧材料时，按以下优先级判断可信度：

1. 用户当前明确指令；
2. 用户长期红线和明确偏好；
3. 真实代码、diff、日志、命令输出、截图、测试结果；
4. 最新被用户确认的 `todolist.md`；
5. 最新被用户确认的 handoff；
6. 旧聊天中的用户确认结论；
7. 旧聊天中的 AI 分析；
8. 旧聊天中的临时猜测、试错和安抚性结论。

低优先级内容不得覆盖高优先级内容。旧 AI 结论与真实证据冲突时，必须剔除 AI 结论。

### 2. 先输出中转蒸馏审核卡

生成文件前，先输出简短审核卡，停止等待用户确认，除非用户明确要求直接生成。

审核卡只判断：

```text
中转蒸馏审核卡：
- 材料是否完整：是 / 否 / 不确定
- 是否适合中转蒸馏：是 / 否
- 是否检测到 TodoList：是 / 否 / 不确定
- 当前材料可迁移证据是否足够生成 handoff-evidence.md：是 / 否 / 不确定
- 新窗口继续工作是否缺关键外部材料：是 / 否 / 不确定
- 需用户补充的外部材料：无 / 列表
- 主要风险：...
- 建议动作：生成 / 生成但最终回答提示补充外部材料 / 先补充材料再生成 / 改用常规 handoff / 本次只做评审
```

审核卡中的两个证据判断必须分开：

- `当前材料可迁移证据是否足够生成 handoff-evidence.md`：判断已上传或粘贴的旧窗口材料中，是否有足够可复制、可摘录、可迁移的文本证据用于证据归档；
- `新窗口继续工作是否缺关键外部材料`：判断新窗口要继续诊断、复核或执行时，是否仍需要重新上传截图、图片、zip、源码包、完整日志、diff、视频画面、本地文件或当前 Prompt 等外部材料。

允许同时出现“可生成 handoff-evidence.md = 是”与“新窗口缺关键外部材料 = 是”。这表示中转材料足够生成交接包，但新工作窗口继续推进仍需用户补充外部原件。不要写成互相矛盾的表达。

### 3. 必须生成完整证据归档

中转蒸馏默认输出 `handoff-evidence.md`。

`handoff-evidence.md` 必须优先保存完整可迁移证据，而不是只写摘要，包括：

- 用户确认结论的原话或完整结论；
- 真实代码、diff、日志、命令输出、截图、测试结果中的关键完整片段；
- 已读取文件、Prompt、Skill、模板的关键原文、完整片段和读取范围；
- 原始材料定位；
- 未完整携带的外部证据、缺口和复核方式；
- 不需要迁移的旧材料。

旧窗口附件、图片、zip、源码包、本地路径指向的文件，不能假设会自动迁移到新窗口。如果这些材料是关键证据，应在生成前或最终回答中提示用户补充，而不是单独新增补证文件。

### 4. 去毒蒸馏

中转窗口必须主动剔除：

- AI 脑补和未证实判断；
- 顺从性、安抚性或临时妥协结论；
- 被用户否定的方案；
- 违反最小改动原则的修复方向；
- 未审计入口就给出的 patch；
- 硬编码字段；
- 为了推进而忽略风险的建议；
- 大段背景铺垫、重复解释、情绪化争执；
- 无后续价值的失败尝试。

但不要删除所有失败路径。会影响后续避坑的废弃方案，应压缩保留为“废弃路径 / 禁区”。

### 5. 关键锚点优先

中转蒸馏必须先找出最高锚点，例如：

- 用户明确红线；
- 当前真实代码状态；
- 当前真实报错；
- 当前真实 diff；
- 当前真实配置语义；
- 当前真实读写入口；
- 用户最新确认的方向；
- 已通过复核的阶段结论；
- 最新可信 `todolist.md` 中的最终目标、当前阶段目标和 P0。

任何旧聊天内容只要与最高锚点冲突，就不能作为新 handoff 或新 TodoList 的事实继承。

### 6. TodoList 只做可信滚动同步

如果旧任务启用了 `todolist.md`：

- 用户提供旧 `todolist.md` 时，按 `skills/04a-goal-todolist.md` 同步输出更新后的 `todolist.md`；
- 旧 handoff 或旧聊天声明有 TodoList，但未提供原文件时，只能记录为证据缺口，不得根据零散聊天重建确定性 TodoList；
- 旧窗口 AI 计划不能自动继承为 TodoList 主线；
- 支撑设施、工具链维护、验证脚本、索引、模板、检查命令等，默认只能作为支撑设施、验收 gate、废弃项或待确认项；
- 下一步原子任务必须来自当前 P0 或当前阶段目标。

如果最终目标、当前阶段目标或 P0 在旧材料中发生漂移或冲突，不得直接生成修正版 TodoList；必须按 `04a-goal-todolist.md` 输出审核卡，除非用户明确要求直接生成且材料中三项锚点可信。

### 7. 单份当前状态快照

最终只输出一份新的：

```text
handoff-current.md
handoff-evidence.md
```

如果任务启用了 TodoList 且用户提供了原文件，同时输出一份更新后的：

```text
todolist.md
```

不要输出多份互相竞争的版本。冲突和不确定项应进入 handoff 的“冲突 / 待确认”、evidence 的“证据缺口 / 待复核”、TodoList 的“待确认 / 候选项”。

### 8. 下一步必须原子化

下一步必须可审计、可执行、边界小。

不应写成：

- “直接修复所有问题”；
- “重构整套流程”；
- “补齐所有测试”；
- “全量改造”。

应写成：

- “先审计真实写入入口清单”；
- “先确认配置读取与写入语义”；
- “先让执行端提供某文件 diff”；
- “先复核某个具体失败现象”。

## 推荐输出结构

### handoff-current.md

```markdown
# Handoff Current

## 0. 适用性判断
## 1. 当前主题
## 2. 最终目标
## 3. 最高锚点
## 4. 当前阶段目标
## 5. 当前状态
## 6. 已确认结论
## 7. 用户偏好与长期规则
## 8. 废弃路径 / 禁区
## 9. 关键文件 / 路径 / 名称
## 10. 证据材料包
## 11. 证据与复核要求
## 12. 冲突 / 待确认
## 13. 配套 TodoList（仅在已启用时保留）
## 14. 下一步原子任务
## 15. 新窗口启动提示
```

### handoff-evidence.md

优先使用 `templates/handoff-evidence-pack.md` 的结构。至少包含：

1. 本文件用途；
2. 完整可迁移证据；
3. 原始材料定位；
4. 未完整携带 / 需要外部补充的证据；
5. 证据缺口 / 待复核；
6. 不需要携带的材料；
7. 新窗口读取建议。

## 输出风格

- 中文；
- 结构清晰；
- 高密度；
- 少背景；
- 删除流水账；
- 明确不确定点；
- 明确废弃路径；
- 下一步必须原子化；
- 多文件优先输出 `.zip`。

## 禁止事项

- 禁止继续业务执行或继续修 Bug。
- 禁止把旧聊天写成摘要流水账。
- 禁止把旧 AI 的猜测、临时验证或顺从性结论写成事实。
- 禁止把已废弃路径包装成可选方案。
- 禁止多轮 handoff 直接拼接导致越滚越大。
- 禁止省略必要的 `handoff-evidence.md`。
- 禁止把 `handoff-evidence.md` 写成只有摘要、没有关键证据正文的索引。
- 禁止把不可迁移、不可复核的材料写成已完整携带证据。
- 禁止为补证清单默认新增独立补证清单文件。
- 禁止把读取 handoff 的启动提示和生成 handoff 的完整规则混在一起。
