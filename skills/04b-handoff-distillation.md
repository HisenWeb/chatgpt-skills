# ChatGPT 专属 Skill：中转窗口去毒蒸馏与 Handoff

## 触发场景

当用户提出以下需求时，启用本 Skill：

- 用户明确说明这是“中转窗口”；
- 用户上传或粘贴完整旧聊天记录；
- 用户要求从旧窗口材料生成 `handoff-current.md`；
- 用户要求对旧窗口进行蒸馏、去毒、压缩、纠偏；
- 旧窗口很长、很乱、跨主题严重；
- 旧窗口中出现模型降智、复读、顺从、胡乱修复、过度设计；
- 需要基于旧 handoff + 完整聊天记录生成新的 `handoff-current.md`；
- 需要把旧窗口的有效结论迁移到新工作窗口。

## 不触发场景

以下情况不要使用本中转蒸馏版，应改用“常规版”：

- 当前窗口主题集中；
- 用户只是普通换窗口；
- 不需要读取完整旧聊天记录；
- 不需要剔除大量错误尝试或上下文污染；
- 当前窗口仍然可靠，直接生成当前状态快照即可。

## 核心定位

本 Skill 用于“三窗口迁移流程”中的中转窗口。

流程如下：

```text
旧工作窗口
  ↓ 导出 / 复制完整聊天记录
中转蒸馏窗口
  ↓ 生成 handoff-current.md
新工作窗口
  ↓ 只读取 handoff-current.md 继续
```

中转窗口的任务不是继续业务讨论，也不是继续修 Bug。

中转窗口只负责：

1. 读取旧材料；
2. 识别有效事实；
3. 剔除污染内容；
4. 合并用户红线；
5. 生成新的 `handoff-current.md`；
6. 给出新窗口启动提示。

中转蒸馏 handoff 的目标是：

> 从完整旧聊天记录中提取下一窗口继续工作所需的最小高保真事实源，同时剔除错误尝试、AI 脑补、顺从性结论、临时妥协和无后续价值的中间过程。

它不是普通摘要。

它是对旧窗口的审计、纠偏、脱水和重组。

最终输出必须是一份新的：

`handoff-current.md`

新工作窗口只读取这份 handoff，不继承旧聊天全文。

## 工作原则

### 1. 输入优先级

处理旧材料时，必须按以下优先级判断可信度：

1. 用户当前明确指令；
2. 用户长期红线和明确偏好；
3. 真实代码、diff、日志、命令输出、截图、测试结果；
4. 最新被用户确认的 handoff；
5. 旧聊天中的用户确认结论；
6. 旧聊天中的 AI 分析；
7. 旧聊天中的临时猜测、试错和安抚性结论。

低优先级内容不得覆盖高优先级内容。

如果旧聊天中的 AI 结论与真实证据冲突，必须剔除 AI 结论。

如果用户当时临时妥协，但该妥协违反用户长期红线，应标记为“废弃路径 / 禁区”，不要继承为新窗口事实。

### 2. 去毒蒸馏

中转窗口必须主动剔除以下内容：

- AI 脑补；
- 未证实判断；
- 顺从性结论；
- 过度迎合用户当时情绪的说法；
- 临时验证方案；
- 被用户否定的方案；
- 违反最小改动原则的修复方向；
- 未审计入口就给出的 patch；
- 硬编码字段；
- 为了推进而忽略风险的建议；
- 大段背景铺垫；
- 重复解释；
- 情绪化争执；
- 无后续价值的失败尝试。

但不要删除所有失败路径。

以下失败路径必须压缩保留：

- 会影响后续避坑的废弃方案；
- 用户明确要求不要再走的路线；
- 已确认会导致代码变脏的方案；
- 已确认与目标冲突的工具路线；
- 已确认不可行的执行方式。

保留形式应为：

```text
废弃路径 / 禁区：
- 方案：...
- 放弃原因：...
- 后续处理：不要再走该路线。
```

### 3. 关键锚点优先

中转蒸馏必须先找出“最高锚点”。

最高锚点包括：

- 用户明确红线；
- 当前真实代码状态；
- 当前真实报错；
- 当前真实 diff；
- 当前真实配置语义；
- 当前真实读写入口；
- 用户最新确认的方向；
- 已通过复核的阶段结论。

任何旧聊天内容只要与最高锚点冲突，就不能作为新 handoff 的事实继承。

### 4. 单份当前状态快照

最终只输出一份新的 handoff。

不要输出多份互相竞争的版本。

不要把旧 handoff 原文拼接进新 handoff。

不要生成“历史日志版 + 精简版 + 执行版”三份文档。

如果存在冲突或不确定项，应放入 handoff 的“冲突 / 待确认”部分，而不是生成多个版本。

### 5. 原子级下一步

中转蒸馏版特别强调：下一步必须可审计、可执行、边界小。

尤其在工程修复场景中，下一步不应写成：

- “直接修复所有问题”；
- “重构整套流程”；
- “补齐所有测试”；
- “全量改造”。

应该写成：

- “先审计真实写入入口清单”；
- “先确认配置读取与写入语义”；
- “先让执行端提供某文件 diff”；
- “先复核某个具体失败现象”；
- “先确认某个字段是否由共享函数统一生成”。

## 输入要求

用户可能提供以下输入：

### 1. 完整旧聊天记录

可能形式：

- Markdown 导出；
- ZIP 包；
- Superpower Chat / GPT-Conversation-Toolkit 导出；
- 手动复制的完整聊天；
- 分段粘贴的旧窗口记录。

### 2. 旧 handoff

可能文件名：

- `handoff-current.md`
- `handoff.md`
- `handoff-项目名.md`

### 3. 用户红线 / 长期规则

例如：

- 最小改动原则；
- 不要 Yes-man；
- 阶段化执行；
- 证据复核；
- 不要硬编码；
- 不要把临时验证正式化；
- 不要工具地狱；
- 长文档优先 `.md` 文件。

### 4. 项目事实证据

例如：

- diff；
- 日志；
- 命令输出；
- 截图；
- 文件内容；
- 目录结构；
- 测试结果；
- 用户明确确认。

## 推荐输出结构

### 0. 适用性判断

说明本次中转蒸馏基于什么材料：

- 完整旧聊天记录；
- 旧 handoff；
- 用户补充红线；
- 上传文件；
- 截图；
- 日志；
- 其他证据。

同时说明可靠性：

- 材料是否完整；
- 是否存在截断；
- 是否存在附件缺失；
- 是否存在证据不足；
- 是否需要执行端补充事实。

### 1. 当前主题

用一两句话说明新窗口要继续的主题。

### 2. 当前目标

说明用户最终想达成什么，而不是旧窗口最后一句话在做什么。

### 3. 最高锚点

列出本 handoff 采用的最高优先级事实和红线。

包括：

- 用户明确红线；
- 当前真实失败现象；
- 当前真实代码 / 配置 / diff 状态；
- 已确认不能违反的原则；
- 最新被用户确认的方向。

### 4. 当前状态

说明已经推进到哪里。

如果是阶段化任务，应写清：

- 当前阶段；
- 已完成阶段；
- 未完成阶段；
- 阻塞点；
- 下一阶段进入条件。

### 5. 已确认结论

列出仍然有效的关键结论。

要求：

- 只写可继承结论；
- 不写无效中间过程；
- 不写旧 AI 的未证实判断；
- 不把临时妥协写成事实。

### 6. 用户偏好与长期规则

只保留与当前主题有关的偏好。

例如：

- 中文表达；
- 结论先行；
- 最小改动；
- 阶段化执行；
- 证据复核；
- 不要 Yes-man；
- 不要工具地狱；
- 不要默认写死具体执行工具；
- 长文档优先生成 `.md` 文件。

### 7. 废弃路径 / 禁区

记录后续不能再走的路线。

每条必须包含：

- 方案或路径；
- 放弃原因；
- 后续处理方式。

### 8. 关键文件 / 路径 / 名称

仅当当前主题涉及项目、仓库、文件或命令时保留。

包括：

- 仓库名；
- 本地路径；
- 文件路径；
- 命令；
- 分支；
- 版本；
- 重要配置文件；
- 关键协议字段；
- 真实读取入口；
- 真实写入入口；
- 共享函数或边界。

如果不涉及项目文件，可以省略。

### 9. 证据与复核要求

说明哪些结论已有证据，哪些还需要证据。

可记录证据类型：

- diff；
- 日志；
- 命令输出；
- 截图；
- 测试结果；
- 文件内容；
- 用户确认。

如果缺少证据，必须写清：

> 待执行端提供证据确认。

### 10. 冲突 / 待确认

记录不能确定、互相冲突或需要新窗口先确认的内容。

不能为了文档漂亮而隐藏不确定性。

### 11. 下一步原子任务

写清新窗口应该从哪里继续。

要求：

- 单任务；
- 小边界；
- 可复核；
- 不要开太多分支；
- 不要要求一次性完成全部工作；
- 不要把已经完成的事情再做一遍。

### 12. 新窗口启动提示

最后提供一段可直接复制到新窗口的启动提示。

启动提示应简洁，只负责让新窗口读取 handoff 并确认理解。

不要把“生成 handoff 的规则”塞进启动提示。

## 输出风格

- 中文；
- 结构清晰；
- 结论先行；
- 高密度；
- 少背景；
- 少铺垫；
- 保留关键细节；
- 删除流水账；
- 明确不确定点；
- 明确废弃路径；
- 下一步必须原子化；
- 优先输出 `.md` 文件。

## 新窗口启动提示模板

### 剪贴板版

适用于用户直接把 handoff 内容粘贴到新窗口。

```text
我们继续上一个窗口的主题。下面是中转窗口生成的 handoff-current.md。

请先阅读并确认你理解：
1. 当前目标；
2. 最高锚点；
3. 当前状态；
4. 已确认结论；
5. 用户偏好与长期规则；
6. 废弃路径 / 禁区；
7. 证据缺口；
8. 下一步原子任务。

要求：
- 不要重新读取旧聊天全文。
- 不要重新发散。
- 不要继承旧窗口中的错误尝试。
- 如果发现 handoff 内部冲突或证据不足，先指出。
- 后续回答必须遵守 handoff 中的工作边界。

handoff-current.md 内容如下：
【粘贴 handoff】
```

### 文件版

适用于用户上传 handoff 文件到新窗口。

```text
我们继续上一个窗口的主题。我已上传中转窗口生成的 handoff-current.md。

请先读取该文件，并用简短要点确认：
1. 当前目标；
2. 最高锚点；
3. 当前状态；
4. 必须遵守的规则；
5. 废弃路径 / 禁区；
6. 未决问题；
7. 下一步原子任务。

确认后再继续工作，不要重新发散，不要继承旧窗口中的错误尝试。
```

## 中转窗口禁止事项

- 禁止继续业务执行。
- 禁止在中转窗口里继续修 Bug。
- 禁止把旧聊天写成摘要流水账。
- 禁止保留无后续价值的中间过程。
- 禁止把旧 AI 的猜测写成事实。
- 禁止把临时验证写成正式方案。
- 禁止因为旧窗口里用户曾短暂妥协，就覆盖用户长期红线。
- 禁止把已废弃路径包装成可选方案。
- 禁止多轮 handoff 直接拼接，导致越滚越大。
- 禁止生成多份互相竞争的 handoff。
- 禁止把“读取 handoff 的启动提示”和“生成 handoff 的完整规则”混在一起。
- 禁止在复杂 handoff 中使用容易复制损坏的嵌套 Markdown 大代码块。
