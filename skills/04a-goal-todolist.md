# ChatGPT 专属 Skill：目标锚定与 TodoList 初始化

## 触发场景

当用户提出以下需求时，启用本 Skill：

- 用户要求生成 `todolist.md`；
- 用户说“方向确定了，帮我列主线 / TodoList / 大方向待办”；
- 当前讨论已经从自由探索进入持续任务；
- 当前任务后续大概率需要跨窗口继续推进；
- 用户发现任务容易漂移，需要建立主线推进账本；
- Handoff Skill 在换窗口时需要同步更新已有 `todolist.md`，并需要引用本 Skill 的 TodoList 语义规则。

## 不触发场景

以下情况不要启用本 Skill：

- 一次性问答；
- 简单解释、翻译、改写；
- 简单命令帮助；
- 临时查资料；
- 没有后续推进需求的小问题；
- 当前任务只是普通 handoff，且没有启用过 `todolist.md`；
- 用户只是要聊天摘要，而不是持续任务推进账本。

## 核心定位

本 Skill 不是通用项目管理系统，也不是普通待办清单生成器。

本 Skill 的目标是：

> 在持续任务进入稳定方向时，创建第一版 `todolist.md`，并定义后续 Handoff 同步 TodoList 时必须遵守的主线锚定规则。

`TodoList` 的职责是防止任务漂移，而不是记录所有细枝末节。

必须区分：

- 最终目标；
- 当前阶段目标；
- 支撑设施 / 验收 gate；
- 大方向 TodoList；
- 下一步原子任务；
- 待确认 / 候选项；
- 已完成 / 暂停 / 废弃项。

## 工作原则

### 0. 必须先做目标锚定预检

生成或更新 `todolist.md` 前，必须先从当前窗口抽取并复述两句话：

```text
最终目标：...
当前阶段目标：...
```

目标锚定优先级：

1. 用户当前明确表达的目标；
2. 当前窗口已经确认的方向；
3. 用户上传或粘贴的最新 handoff / todolist；
4. 相关 Skill / Template 的规则；
5. README、SKILL_INDEX、doctor、脚本、模板等仓库维护上下文；
6. AI 自己的推断。

如果无法从 1-3 确认最终目标，不能继续生成确定性 TodoList；必须把不确定点放入“待确认 / 候选项”，或先询问用户。

禁止从“当前启用的 Skill 名称”“仓库维护规则”“doctor / README / SKILL_INDEX / 模板”等支撑设施倒推出最终目标。

### 1. 首次初始化优先

本 Skill 最适合用于首次创建 `todolist.md`。

触发时机通常是：

```text
自由讨论 / 方向探索
↓
方向基本确定
↓
创建 todolist.md
```

首次 `todolist.md` 不要求完美，但必须明确最终目标、当前阶段目标、大方向队列和下一步原子任务。

### 2. TodoList 是主线推进账本

`todolist.md` 负责记录持续任务的主线推进。

它不是：

- 文件修改清单；
- 命令执行清单；
- doctor / README / 索引维护清单；
- 聊天流水账；
- 普通备忘录；
- 项目管理系统；
- 覆盖矩阵；
- 仓库治理计划。

### 3. 支撑设施不能变成最终目标或 P0 主线

如果任务中存在 README、索引、模板、doctor、测试脚本、检查命令等内容，必须把它们放在“支撑设施 / 验收 gate”中。

禁止把支撑设施维护提升为大方向 TodoList 的主线目标，除非用户明确说明当前阶段目标就是维护这些设施。

P0 必须直接服务用户确认的最终目标。默认情况下，P0 应该是“用户最终要拿到的交付物”或“直接阻塞该交付物的当前阶段动作”，而不是仓库结构、索引、脚本、覆盖检查、文档同步等维护项。

### 4. 大方向 TodoList 与下一步原子任务必须分离

`大方向 TodoList` 是阶段级 / 方向级待办队列。

`下一步原子任务` 是从当前 P0 或当前阶段目标中拆出的一个最小动作。

禁止用“下一步建议”替代 TodoList。

禁止把 TodoList 写成大量细碎操作。

下一步原子任务必须满足：

- 只能写一个；
- 必须来自当前 P0 或当前阶段目标；
- 必须能推进用户确认的最终目标；
- 不能从支撑设施、候选项或 AI 推断方向里拆出，除非用户明确要求。

### 5. 新增 Todo 项必须有准入理由

新增大方向 Todo 项必须满足至少一个条件：

- 直接服务于最终目标；
- 是当前阶段目标的必要后续阶段；
- 解决已确认阻塞；
- 用户明确新增了方向；
- 旧 TodoList 漏掉了已经确认的重要方向。

AI 自己推测出来的新方向，默认只能进入“待确认 / 候选项”，不能自动成为 P0 主线。

### 6. 状态更新必须有依据

Handoff 同步 `todolist.md` 时，只能基于以下依据更新状态：

- 用户明确确认；
- 当前窗口明确结论；
- 真实代码、diff、日志、命令输出、截图、测试结果；
- 已通过复核的阶段结果。

证据不足时，必须标为“待确认”，不能为了文档整洁强行标完成。

### 7. 反漂移示例必须内化

如果用户当前最终目标是：

```text
搞几个 ChatGPT 专属 Skill 开头声明
```

那么正确主线应围绕：

- 确认第一批核心 Skill；
- 编写这些 Skill 的开头声明；
- 复核它们是否能解决用户真实痛点；
- 决定是否正式入仓库。

错误主线包括：

- 做仓库覆盖矩阵审计；
- 先治理 README / SKILL_INDEX / doctor；
- 把模板、索引、脚本维护写成 P0；
- 把“建设完整 Skill 体系”替代“先产出几个专属 Skill 声明”。

如果发现上一版 TodoList 已经出现这种漂移，必须先暂停生成结果，指出源头规则缺口，再给出修正规则或修正版 TodoList。

### 8. 换窗口时由 Handoff Skill 同步更新

本 Skill 不要求用户每次手动调用来更新 TodoList。

推荐关系是：

```text
首次方向确定：
  用本 Skill 创建 todolist.md

准备换窗口：
  用 Handoff Skill 生成 handoff-current.md
  同时输入旧 todolist.md
  Handoff Skill 按本 Skill 的规则同步更新 todolist.md
```

也就是说，换窗口时 Handoff Skill 是主流程，本 Skill 提供 TodoList 的结构和同步规则。

## 输入要求

首次初始化时，输入通常是：

- 当前窗口讨论内容；
- 用户确认的最终目标或方向；
- 当前阶段边界；
- 已知支撑设施；
- 用户禁止事项；
- 已知证据缺口或待确认问题。

换窗口同步时，输入通常是：

- 当前窗口上下文；
- 旧 `todolist.md`；
- 当前 handoff 生成要求；
- 用户最新确认或证据。

如果输入中同时包含“业务目标”和“Skill / Template / README / doctor 等仓库维护上下文”，必须先用目标锚定优先级区分二者，不能混成同一个目标。

## 输出要求

首次初始化时，输出文件名默认是：

```text
todolist.md
```

推荐结构：

```markdown
# TodoList / 主线推进账本

## 0. 目标锚定预检

- 最终目标：
- 当前阶段目标：
- 本次是否需要 todolist.md：
- 漂移风险：

## 1. 最终目标

说明这个持续任务最终要达成什么。

## 2. 当前阶段目标

说明当前阶段只解决什么，不解决什么。

## 3. 支撑设施 / 验收 gate

列出 README、索引、模板、doctor、测试、复核命令等支撑设施。

说明：这些是 gate，不是主线目标。

## 4. 大方向 TodoList

- [P0][进行中] ...
  - 目标：
  - 完成标准：
  - 依赖 / 阻塞：

- [P1][待开始] ...
  - 目标：
  - 完成标准：
  - 依赖 / 阻塞：

## 5. 下一步原子任务

只能写一个当前最该做的小动作。

## 6. 待确认 / 候选项

AI 认为可能要加，但还没有被用户确认的项。

## 7. 已完成 / 暂停 / 废弃

只保留对后续避坑有价值的完成项或废弃项。
```

换窗口同步时，应输出：

1. `handoff-current.md`；
2. 更新后的 `todolist.md`。

`handoff-current.md` 只引用和摘要 TodoList，不完整复制 TodoList 正文。

## 状态与优先级

推荐优先级：

```text
[P0] 当前必须先处理
[P1] 主线后续
[P2] 可延后优化
```

推荐状态：

```text
[待开始]
[进行中]
[已完成]
[阻塞]
[待确认]
[暂停]
[废弃]
```

不要在第一版中加入复杂负责人、截止日期、评分、依赖图等项目管理字段，除非用户明确要求。

## 与 Handoff 的关系

`handoff-current.md` 是上下文快照。

`todolist.md` 是主线推进账本。

两者关系：

```text
首次方向确定：
  TodoList Skill 创建 todolist.md

准备换窗口：
  Handoff Skill 读取当前窗口和旧 todolist.md
  输出 handoff-current.md + 更新后的 todolist.md

新窗口启动：
  同时读取 handoff-current.md 和 todolist.md
  按 todolist.md 推进主线
  按 handoff-current.md 遵守上下文、证据和边界
```

如果 handoff 与 TodoList 冲突，新窗口必须先指出冲突，不要自行合并。

## 禁止事项

- 禁止把 TodoList 写成聊天摘要。
- 禁止把 TodoList 写成命令清单。
- 禁止把 TodoList 写成文件修改清单。
- 禁止把支撑设施维护写成最终目标。
- 禁止把支撑设施维护写成 P0 主线。
- 禁止用“下一步建议”替代大方向 TodoList。
- 禁止让 AI 自己脑补新主线。
- 禁止把待确认项自动升级为 P0。
- 禁止从 Skill 名称、模板名称、README、SKILL_INDEX、doctor 等反推出最终目标。
- 禁止把覆盖矩阵、仓库治理、索引同步默认写成当前阶段目标。
- 禁止删除旧项而不说明原因。
- 禁止为了整洁隐藏冲突或证据缺口。
- 禁止把本 Skill 扩展成复杂项目管理系统。
