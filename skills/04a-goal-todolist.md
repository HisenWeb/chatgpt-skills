# ChatGPT 专属 Skill：极简主线 TodoList 初始化

## 触发场景

当用户明确提出以下需求时，启用本 Skill：

- 生成、初始化、修正或更新 `todolist.md`；
- 建立“主线 TodoList”“大方向 TodoList”“防漂移 TodoList”；
- 用户明确要求“把这些事项当成 TodoList”；
- 用户提供了旧 `todolist.md`，并要求继续、同步、换窗口或滚动更新；
- Handoff Skill 检测到已存在 `todolist.md`，需要按本 Skill 的语义同步状态。

注意：不要仅因为“任务较长”“可能跨窗口”“正在讨论多个方向”就自动启用本 Skill；必须有明确的 TodoList 需求或已存在的 `todolist.md`。

## 不触发场景

以下情况不要启用本 Skill：

- 一次性问答；
- 简单解释、翻译、改写；
- 临时查资料；
- 单次命令帮助；
- 普通聊天摘要；
- 普通 handoff，且未检测到已启用的 `todolist.md`；
- 用户只是讨论长期方向，但没有要求建立或同步 TodoList；
- 当前任务只有支撑设施维护，没有用户确认的主线事项。

## 核心定位

本 Skill 不是通用项目管理系统，也不是复杂任务规划器。

本 Skill 的目标是：

> 用最小 checkbox 清单固定持续任务的主线事项，防止 TodoList 被写成聊天摘要、文件清单、命令清单、阶段报告或 AI 自己发散出的计划。

`TodoList` 只回答一个问题：

> 后续主线要推进哪几个大事项？

默认格式必须极简：

```markdown
# TodoList

- [ ] 主线事项一
- [ ] 主线事项二
- [ ] 主线事项三
```

除非用户明确要求扩展，禁止加入“最终目标、当前阶段目标、当前 P0、支撑设施、验收 gate、下一步原子任务、候选项、已废弃项”等复杂结构。

## 总流程

```text
用户请求创建 / 更新 TodoList
↓
判断是否真的需要 todolist.md
↓
提取用户已确认的主线事项
↓
事项清楚：直接输出极简 checkbox TodoList
事项不清楚：先输出简短候选清单，请用户确认
↓
用户确认 / 修正
↓
生成或同步正式 todolist.md
```

## 工作原则

### 1. 必须先判断是否需要 TodoList

生成 `todolist.md` 前，必须先判断当前任务是否真的需要持续主线清单。

如果只是一次性问答、临时资料查询、简单解释、单次命令帮助或普通聊天摘要，必须直接说明“不需要 `todolist.md`”，不要强行初始化。

### 2. TodoList 只写主线事项

TodoList 条目必须是用户要持续推进的大方向事项或阶段性交付物。

适合写入：

- 一个待创建 / 待完善的 Skill；
- 一个明确阶段交付物；
- 一个用户已确认需要持续推进的主线方向；
- 一个会跨窗口继续推进的核心事项。

不适合写入：

- 命令执行步骤；
- 文件修改清单；
- README / 索引 / doctor 等支撑设施，除非用户明确把它们作为主线事项；
- “先讨论一下”“整理一下”“生成方案”等过程动作；
- AI 自己推断出来但用户尚未确认的新方向。

### 3. 不使用 P0 / P1 / 阶段目标结构

本 Skill 默认不使用 P0 / P1 / P2，不生成复杂优先级结构。

如果用户要求“先做哪个”，可以在聊天回复里说明当前建议，但不要把 `todolist.md` 扩展成项目管理文档。

### 4. 事项不清楚时只做候选清单

如果主线事项不清楚，不要生成正式 `todolist.md`。

只输出极简候选清单：

```markdown
待确认 TodoList：

- [ ] 候选事项一
- [ ] 候选事项二

请确认这些是否就是 TodoList 主线项。
```

候选清单不是正式 `todolist.md`。

### 5. 用户给出的清单优先

如果用户明确给出类似以下内容：

```markdown
- [ ] A
- [ ] B
- [ ] C
```

并说明“这就是 TodoList”，必须以该清单为准，不要再改写成复杂结构。

可以只做必要的错字修正或格式统一，不得擅自添加目标、P0、候选项或说明段落。

### 6. Handoff 同步时只同步状态和必要新增项

换窗口同步 `todolist.md` 时，只允许做三类动作：

- 根据用户明确确认或可靠证据，把 `[ ]` 改成 `[x]`；
- 根据用户明确新增的主线方向，新增一条 checkbox；
- 根据用户明确废弃的方向，删除该条或保留为 `[x]` 后在聊天中说明。

证据不足时不要把事项标完成。

## 输出要求

### 正式 todolist.md

默认只输出：

```markdown
# TodoList

- [ ] ...
- [ ] ...
```

如已有完成项，可以使用：

```markdown
# TodoList

- [x] 已完成事项
- [ ] 未完成事项
```

### 聊天回复

生成文件前后的聊天回复必须简短，只说明：

- 是否需要 `todolist.md`；
- 输出的是正式 TodoList 还是待确认候选清单；
- 是否需要用户确认。

不要把回复写成小型 handoff 或长篇规划。

## 与 Handoff 的关系

`handoff-current.md` 是上下文快照。

`todolist.md` 是极简主线清单。

两者关系：

```text
首次方向确定：
本 Skill 生成极简 todolist.md，或先输出候选清单等用户确认

准备换窗口：
由对应 Handoff Skill 判断是否输出 handoff-current.md
如已确认需要同步 todolist.md，则按本 Skill 只同步主线事项和 checkbox 状态

新窗口启动：
同时读取 handoff-current.md 和 todolist.md
按 handoff-current.md 理解背景
按 todolist.md 识别主线事项
```

如果 handoff 与 TodoList 冲突，新窗口必须先指出冲突，不要自行合并。

## 禁止事项

- 禁止仅因任务较长或可能跨窗口就自动创建 TodoList。
- 禁止把 TodoList 写成聊天摘要、命令清单、文件修改清单或检查矩阵。
- 禁止默认加入“最终目标、当前阶段目标、当前 P0、支撑设施、验收 gate、下一步原子任务、待确认候选、已废弃项”等复杂结构。
- 禁止把支撑设施维护默认写成主线事项。
- 禁止把 AI 自己推断的新方向自动加入正式 TodoList。
- 禁止为了显得完整而扩写用户给出的极简清单。
- 禁止把本 Skill 扩展成复杂项目管理系统。
- 禁止为了格式整齐隐藏冲突或证据缺口。
