# ChatGPT 专属 Skill：目标锚定与 TodoList 初始化

## 触发场景

当用户明确提出以下需求时，启用本 Skill：

- 生成、初始化、修正或更新 `todolist.md`；
- 建立“主线推进账本”“大方向 TodoList”“防漂移 TodoList”；
- 用户提供了旧 `todolist.md`，并要求继续、同步、换窗口或滚动更新；
- Handoff Skill 在已检测到 `todolist.md` 时，需要引用本 Skill 的 TodoList 语义规则；
- 用户明确要求用 TodoList 固定持续任务的最终目标、当前阶段目标和 P0。

注意：不要仅因为“任务较长”“可能跨窗口”“正在讨论多个方向”就自动启用本 Skill；必须有明确的 TodoList 需求或已存在的 `todolist.md`。

## 不触发场景

以下情况不要启用本 Skill：

- 一次性问答；
- 简单解释、翻译、改写；
- 临时查资料；
- 单次命令帮助；
- 普通聊天摘要；
- 普通 handoff，且未检测到已启用的 `todolist.md`；
- 用户只是讨论长期方向，但没有要求建立或同步 TodoList；
- 当前任务只有支撑设施维护，没有用户确认的持续主线目标。

## 核心定位

本 Skill 不是通用项目管理系统，也不是普通待办清单生成器。

本 Skill 的目标是：

> 用最小结构固定持续任务的主线锚点，防止 TodoList 被写成聊天摘要、文件清单、工具治理计划或 AI 自己发散出的新路线。

`TodoList` 的职责是防止任务漂移，而不是记录所有细枝末节。

必须区分：

- 最终目标；
- 当前阶段目标；
- 当前 P0；
- 支撑设施 / 验收 gate；
- 大方向 TodoList；
- 下一步原子任务；
- 待确认 / 候选项；
- 已完成 / 暂停 / 废弃项。

## 总流程

```text
用户请求创建 / 更新 TodoList
↓
判断是否真的需要 todolist.md
↓
目标锚定预检：最终目标 / 当前阶段目标 / 当前 P0
↓
首次初始化默认输出审核卡，等待用户确认
↓
用户确认 / 修正
↓
生成或同步正式 todolist.md
```

如果已经存在有效 `todolist.md`，可做滚动同步；但只要最终目标、当前阶段目标或 P0 发生实质变化，必须先按审核卡重新确认，不能直接改写主线。

## 工作原则

### 0. 必须先判断是否需要 TodoList

生成审核卡或 `todolist.md` 前，必须先判断当前任务是否真的需要持续主线账本。

如果只是一次性问答、临时资料查询、简单解释、单次命令帮助或普通聊天摘要，必须直接说明“不需要 `todolist.md`”，不要强行初始化。

### 1. 必须先做目标锚定预检

生成审核卡、创建 `todolist.md` 或更新 `todolist.md` 前，必须先抽取并复述三项：

```text
最终目标：...
当前阶段目标：...
当前 P0：...
```

目标锚定优先级：

1. 用户当前明确表达的目标；
2. 用户明确确认过的最新目标 / 阶段 / P0；
3. 用户上传或粘贴的最新可信 handoff / `todolist.md`；
4. 当前窗口中有明确证据支撑的阶段结论；
5. 相关 Skill / Template 的规则；
6. AI 自己的推断。

如果无法从 1-4 确认最终目标、当前阶段目标或 P0，不能生成确定性 TodoList；必须把不确定项放入审核卡或“待确认 / 候选项”。

禁止从“当前启用的 Skill 名称”“模板名称”“仓库结构”“检查脚本”“索引文件”“工具链维护上下文”等支撑设施倒推出最终目标。

### 2. 首次初始化默认必须经过用户审核 Gate

首次创建 `todolist.md` 时，默认先输出「todolist 生成前审核卡」，并停止等待用户确认。

审核卡目标不是说明背景，而是让用户快速判断三件事是否正确：

- 最终目标；
- 当前阶段目标；
- 当前 P0。

审核卡必须短，不写成小型 handoff。默认不超过 12 行，不展开证据报告，不生成完整 TodoList，不提供下载文件。

审核卡默认使用状态符号：

```text
✓ 已确认 / 方向基本明确
◐ 当前焦点 / 需要用户确认
○ 待确认候选
✕ 不进入主线 / 已排除
```

只有当用户明确确认审核卡，或修正三项后要求生成，才允许生成正式 `todolist.md`。

### 3. 可跳过审核 Gate 的情况

满足以下条件时，可以跳过审核卡并直接生成正式 `todolist.md`：

1. 用户明确要求“直接生成正式 `todolist.md`”或同等含义；
2. 输入材料中已经包含可信的最终目标、当前阶段目标和当前 P0；
3. 可信来源来自用户当前明确表达、用户确认过的 handoff / `todolist.md`、或当前窗口中清楚可追溯的确认结论；
4. 生成文件时必须在“目标锚定预检”中写明跳过审核 Gate 的依据。

如果三项锚点缺失、互相冲突，或只是 AI 推断，一律不能跳过审核卡。

### 4. TodoList 是主线推进账本

`todolist.md` 负责记录持续任务的主线推进。

它不是：

- 聊天流水账；
- 文件修改清单；
- 命令执行清单；
- 支撑设施维护清单；
- 检查矩阵；
- 普通备忘录；
- 复杂项目管理系统。

### 5. 支撑设施不能替代用户目标

文档、索引、模板、脚本、配置、检查命令、测试、验收 gate、工具链维护等内容，默认属于“支撑设施 / 验收 gate”。

它们可以进入 TodoList 的支撑设施区，也可以作为某个 Todo 项的完成标准或依赖，但不能默认替代用户最终目标、当前阶段目标或 P0。

只有当用户明确说明“当前阶段目标就是维护这些设施”时，才可以把对应设施维护写入主线。

### 6. 大方向 TodoList 与下一步原子任务必须分离

`大方向 TodoList` 是阶段级 / 方向级待办队列。

`下一步原子任务` 是从当前 P0 或当前阶段目标中拆出的一个最小动作。

禁止用“下一步建议”替代 TodoList。

禁止把 TodoList 写成大量细碎操作。

下一步原子任务必须满足：

- 只能写一个；
- 必须来自当前 P0 或当前阶段目标；
- 必须能推进用户确认的最终目标；
- 不能从支撑设施、候选项或 AI 推断方向里拆出，除非用户明确要求。

### 7. 新增 Todo 项必须有准入理由

新增大方向 Todo 项必须满足至少一个条件：

- 直接服务于最终目标；
- 是当前阶段目标的必要后续阶段；
- 解决已确认阻塞；
- 用户明确新增了方向；
- 旧 TodoList 漏掉了已经确认的重要方向。

AI 自己推测出来的新方向，默认只能进入“待确认 / 候选项”，不能自动成为 P0 主线。

### 8. 状态更新必须有依据

Handoff 同步 `todolist.md` 时，只能基于以下依据更新状态：

- 用户明确确认；
- 当前窗口明确结论；
- 真实代码、diff、日志、命令输出、截图、测试结果；
- 已通过复核的阶段结果。

证据不足时，必须标为“待确认”，不能为了文档整洁强行标完成。

### 9. 反漂移规则必须抽象执行

如果发现 TodoList 正在从“用户要达成的目标”漂移到“工具、文档、索引、脚本、测试矩阵、流程治理”等支撑设施，必须先暂停生成，指出漂移源头，再输出审核卡或按用户确认修正。

判断标准：

- 正确主线应围绕用户最终要拿到的结果、当前阶段交付物、直接阻塞项和验收标准；
- 支撑设施只能服务主线，不能自动成为主线；
- 候选方向不能自动变成 P0；
- 旧窗口 AI 计划不能自动继承为 TodoList 主线。

### 10. 换窗口时由 Handoff Skill 同步更新

本 Skill 不要求用户每次手动调用来更新 TodoList。

推荐关系是：

```text
首次方向确定：
本 Skill 生成审核卡，用户确认后创建 todolist.md

准备换窗口：
Handoff Skill 读取当前窗口和旧 todolist.md
按本 Skill 的规则同步更新 TodoList
```

也就是说，换窗口时 Handoff Skill 是主流程，本 Skill 提供 TodoList 的结构和同步语义。

如果换窗口同步时发现最终目标、当前阶段目标或 P0 发生实质变化，不得直接改写 TodoList 主线，必须先输出审核卡并等待用户审核；用户明确要求直接生成且材料中三项锚点可信时，才可按本 Skill 的跳过 Gate 条件处理。

## 输入要求

首次初始化时，输入通常是：

- 当前窗口讨论内容；
- 用户确认的最终目标或方向；
- 当前阶段边界；
- 当前 P0；
- 已知支撑设施；
- 用户禁止事项；
- 已知证据缺口或待确认问题。

换窗口同步时，输入通常是：

- 当前窗口上下文；
- 旧 `todolist.md`；
- handoff 生成要求；
- 用户最新确认或证据。

如果输入中同时包含“用户目标”和“支撑设施维护上下文”，必须先按目标锚定优先级区分二者，不能混成同一个目标。

## 输出要求

### 首次初始化：先输出审核卡

首次初始化时，默认输出不是 `todolist.md`，而是短审核卡：

```text
# todolist 生成前审核卡
状态：待确认
✓/◐ 最终目标：...
✓/◐ 当前阶段目标：...
◐ 当前 P0：...
✕ 不进入主线：支撑设施 / 已废弃 / 无关项 ...
○ 待确认候选：...
建议：生成 / 不生成 todolist.md（一句话原因）
请确认：最终目标、当前阶段目标、当前 P0 是否正确？
```

审核卡不应伪装成正式 TodoList，不应包含完整大方向 TodoList，不应生成 `todolist.md` 下载文件。

### 用户确认后：生成正式 todolist.md

用户确认后，输出文件名默认是：

```text
todolist.md
```

推荐结构：

```markdown
# TodoList / 主线推进账本

## 0. 目标锚定预检

- 最终目标：
- 当前阶段目标：
- 当前 P0：
- 本次是否需要 todolist.md：
- 用户审核 Gate：已确认 / 已跳过（说明依据）
- 漂移风险：

## 1. 最终目标

说明这个持续任务最终要达成什么。

## 2. 当前阶段目标

说明当前阶段只解决什么，不解决什么。

## 3. 支撑设施 / 验收 gate

列出文档、索引、模板、脚本、配置、检查命令、测试、复核要求等支撑设施。

说明：这些是 gate 或依赖，不是默认主线目标。

## 4. 大方向 TodoList

- [P0][进行中] ...
  - 目标：
  - 完成标准：
  - 依赖 / 阻塞：

- [P1][待开始] ...
  - 目标：
  - 完成标准：
  - 依赖 / 阻塞：

## 5. 下一步原子任务

只能写一个当前最该做的小动作。

## 6. 待确认 / 候选项

AI 认为可能要加，但还没有被用户确认的项。

## 7. 已完成 / 暂停 / 废弃

只保留对后续避坑有价值的完成项或废弃项。
```

### 换窗口同步时

是否输出 `handoff-current.md` 由对应 Handoff Skill 决定。

本 Skill 只规定：如果 Handoff Skill 已确认需要同步 `todolist.md`，则更新后的 `todolist.md` 必须遵守本 Skill 的目标锚定、审核 Gate、P0、状态同步和反漂移规则。

`handoff-current.md` 可以引用和摘要 TodoList，但不要完整复制 TodoList 正文。

## 状态与优先级

推荐优先级：

```text
[P0] 当前必须先处理
[P1] 主线后续
[P2] 可延后优化
```

推荐状态：

```text
[待开始]
[进行中]
[已完成]
[阻塞]
[待确认]
[暂停]
[废弃]
```

不要在第一版中加入复杂负责人、截止日期、评分、依赖图等项目管理字段，除非用户明确要求。

## 与 Handoff 的关系

`handoff-current.md` 是上下文快照。

`todolist.md` 是主线推进账本。

两者关系：

```text
首次方向确定：
TodoList Skill 先生成审核卡
用户确认后创建 todolist.md

准备换窗口：
由对应 Handoff Skill 判断是否输出 handoff-current.md
如已确认需要同步 todolist.md，则按本 Skill 规则生成更新后的 todolist.md

新窗口启动：
同时读取 handoff-current.md 和 todolist.md
按 todolist.md 推进主线
按 handoff-current.md 遵守上下文、证据和边界
```

如果 handoff 与 TodoList 冲突，新窗口必须先指出冲突，不要自行合并。

## 禁止事项

- 禁止仅因任务较长或可能跨窗口就自动创建 TodoList。
- 禁止首次初始化时直接生成正式 `todolist.md`，除非满足跳过 Gate 条件。
- 禁止绕过用户审核 Gate。
- 禁止把审核卡写成正式 TodoList。
- 禁止把审核卡写成小型 handoff 或长篇说明文档。
- 禁止在用户确认前把 AI 推断方向升级为 P0。
- 禁止把 TodoList 写成聊天摘要、命令清单或文件修改清单。
- 禁止把支撑设施维护默认写成最终目标或 P0 主线。
- 禁止用“下一步建议”替代大方向 TodoList。
- 禁止让 AI 自己脑补新主线。
- 禁止把待确认项自动升级为 P0。
- 禁止从 Skill 名称、模板名称、仓库结构、索引、检查脚本、工具链维护上下文反推出最终目标。
- 禁止删除旧项而不说明原因。
- 禁止为了整洁隐藏冲突或证据缺口。
- 禁止把本 Skill 扩展成复杂项目管理系统。
