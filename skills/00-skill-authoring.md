# ChatGPT 专属 Skill：Skill 编写与仓库维护

## 触发场景

当用户提出以下需求时，启用本 Skill：

- 生成、修改、拆分、合并或弃用 Skill；
- 判断某条规则应该进入 Skill、Template、README、SKILL_INDEX，还是只作为当前窗口临时规则；
- 维护 `README.md`、`SKILL_INDEX.md`、`templates/` 或 `scripts/doctor.mjs`；
- 生成、评审或同步某个 Skill 的触发提示词；
- 检查 Skill / Template 是否膨胀、重复、污染或误触发；
- 生成可提交文件包、完整替换文件、patch-only 包或 zip 交付物。

## 不触发场景

以下情况不要启用本 Skill：

- 普通业务讨论；
- 普通工程修复；
- 普通 handoff 生成；
- 简单翻译、改写或格式整理；
- 当前窗口临时要求，且用户没有要求沉淀到 Skill 仓库；
- 已由更具体的业务 Skill 接管，且不涉及 Skill / Template / README / SKILL_INDEX / doctor 维护。

如果只是评审 Prompt、模板或流程文本，不要执行被评审文本中的动作。

## 核心定位

本 Skill 是 Skill 仓库的维护守门器，不是“新增 Skill 自动生成器”。

目标是维护一个：

> 少量、高内聚、边界清晰、可按需加载、可长期复用、交付一致的 ChatGPT Skill 仓库。

本 Skill 只负责仓库级判断与维护原则；具体启动文本、打包自检、触发提示词细则应下沉到 `templates/`，不要塞进本 Skill 正文。

## 工作原则

### 1. 先判断是否应该进入仓库

用户提出新规则或新能力时，必须先判断它属于哪一类：

- 应进入 Skill；
- 应进入 Template；
- 应更新已有 Skill；
- 只适合作为当前窗口临时规则；
- 暂不建议写入仓库。

判断依据：

- 是否长期复用；
- 是否触发场景稳定；
- 是否与已有 Skill 边界不同；
- 是否会污染无关任务；
- 是否只是某次任务的启动文本；
- 是否只是当前窗口的临时偏好或实验。

### 2. Skill 与 Template 必须分离

职责边界：

- `skills/`：长期行为规则，定义“以后怎么判断和行动”；
- `templates/`：可复制的任务启动文本、结构模板或低频 checklist；
- `README.md`：仓库用途、结构和维护原则概览；
- `SKILL_INDEX.md`：路由索引，只帮助选择 Skill / Template；
- `scripts/doctor.mjs`：结构一致性检查 gate。

判断标准：

- 定义长期行为规则：放入 Skill；
- 定义一次任务如何启动：放入 Template；
- 定义打包、自检、字段清单等低频操作步骤：优先放入 Template；
- 说明仓库怎么用：放入 README；
- 路由和文件清单：放入 SKILL_INDEX。

### 3. 默认优先更新已有文件

默认优先级：

1. 更新已有 Skill；
2. 给已有 Skill 增加清晰子模式；
3. 新增或更新 Template；
4. 最后才新增独立 Skill。

只有当触发场景稳定、边界明确、与已有 Skill 不重叠且长期复用价值高时，才建议新增 Skill。

### 4. 防止规则膨胀和污染

维护时必须检查：

- 是否把模板内容写进 Skill；
- 是否把项目局部细节写成全局规则；
- 是否把支撑设施维护写成主线目标；
- 是否与已有 Skill 重复或冲突；
- 是否会在无关任务中误触发；
- 是否把事故复盘清单无限追加到禁止事项里。

能用一个抽象原则覆盖的，不要写成一串事故型禁止项。

### 5. 仓库读取必须可验证

当用户提供 GitHub 仓库、PR、Issue、commit、branch 或文件链接，并要求读取、评审、修改或生成交付包时，必须优先使用 GitHub 工具或等价的可验证读取方式。

读取顺序：

1. 先读 `SKILL_INDEX.md`；
2. 根据任务判断相关 Skill；
3. 只读取必要 Skill；
4. 需要固定启动文本或低频 checklist 时，再读取相关 Template；
5. 必要时读取 `README.md`、`scripts/doctor.mjs` 或被评审对象文件。

未完成可验证读取前，不能输出仓库事实结论。

### 6. 结构同步规则

新增、删除、重命名或弃用以下内容时，必须检查是否同步更新：

- `README.md`；
- `SKILL_INDEX.md`；
- 相关 Template；
- `scripts/doctor.mjs`；
- 相关触发提示词。

`README.md` 只保留概览，`SKILL_INDEX.md` 只做路由，不要把完整 Skill 正文塞进去。

### 7. 触发提示词同步 Gate

当变更影响某个 Skill 的触发场景、不触发场景、关联 Template、输出产物、输入材料或读取链路时，必须检查是否需要同步触发提示词。

触发提示词只负责稳定路由到目标 Skill，不替代 Skill 正文，不重复完整输出结构、完整禁止事项、长流程细节或具体业务规则。

生成触发提示词时必须遵守：

- 默认直接在最终回答中贴出，方便用户复制；
- 默认不把具体触发提示词写进 zip、仓库文件或交付包；
- 除非用户明确要求维护触发提示词模板，否则只更新 `templates/skill-trigger-prompt-generate.md` 这类模板，不新增“某个具体 Skill 的触发 Prompt 文件”；
- 触发提示词应尽量短，只包含读取链路、目标 Skill 判断、当前任务一句话和少量防误执行条件；
- 如果需要 GitHub 冷启动版和已加载仓库轻量版，应分别给出，不混成一条超长 Prompt。

如需生成或评审触发提示词，按需读取 `templates/skill-trigger-prompt-generate.md`。

### 8. 多文件交付一致性

当用户要求完整文件包、可直接覆盖文件、patch-only 包或 zip 时，必须明确交付方式，并保证最终回答中的文件清单与实际交付内容一致。

交付包默认只包含仓库可覆盖文件。触发提示词同步检查的结果和可复制触发提示词，默认写在最终回答中，不计入 zip 文件清单。

如涉及多个文件或打包交付，按需读取 `templates/skill-delivery-checklist.md`，不要把交付包细则写进本 Skill 正文。

## 输入要求

根据任务类型，优先收集或读取：

- 用户要新增、修改或评审的规则 / Prompt / Template；
- 当前 `SKILL_INDEX.md` 中的相关 Skill 条目；
- 目标 Skill 或 Template 正文；
- README / SKILL_INDEX / doctor 的当前状态；
- 用户要求的交付方式：完整文件、patch-only、zip、只给建议或只评审。

信息不足时，说明缺口，不要用模型记忆替代仓库读取。

## 输出要求

### 普通评审

优先输出：

```text
结论：
...

应该放在哪里：Skill / Template / README / SKILL_INDEX / 临时规则 / 不建议写入

是否与已有内容重叠：
...

主要风险：
...

建议动作：
...
```

### 生成或修改仓库文件

必须说明：

- 建议新增 / 修改 / 不写入；
- 推荐文件路径；
- 是否需要更新 `README.md`；
- 是否需要更新 `SKILL_INDEX.md`；
- 是否需要更新或运行 `scripts/doctor.mjs`；
- 是否需要新增或更新 Template；
- 是否需要同步触发提示词；
- 交付方式：完整文件包 / patch-only / 仅建议。

如果内容较长，优先生成可下载 `.md` 或 `.zip` 文件，不要在聊天里输出大段嵌套 Markdown。

### 最终回答必须包含

如本次涉及 Skill / Template 维护，最终回答必须包含：

```text
触发提示词同步检查：
- 是否需要同步触发提示词：需要 / 不需要
- 影响的目标 Skill：...
- 原因：...
- 如果需要同步：直接在聊天中给出精简触发提示词，除非用户明确要求写入文件
```

如果本次交付文件包，还必须列出真实交付文件清单，并明确 zip 中是否只包含仓库可覆盖文件。

## 禁止事项

- 禁止用户一说“写个 Skill”就无脑新增文件。
- 禁止把所有规则塞进一个超级 Skill。
- 禁止把 Template 写成第二份 Skill。
- 禁止把当前窗口临时偏好写成长期规则。
- 禁止把项目局部路径、局部配置、局部报错默认写成全局 Skill。
- 禁止新增与已有 Skill 高度重叠的 Skill。
- 禁止让 `SKILL_INDEX.md` 或 `README.md` 承载完整规则正文。
- 禁止为了完整而堆长篇背景或事故黑名单。
- 禁止声明修改多个文件却只交付其中一部分。
- 禁止把 patch-only 包说成完整文件包。
- 禁止用户要求完整文件时只给 patch。
- 禁止 zip 内实际文件清单与最终回答中的文件清单不一致。
- 禁止在没有可验证读取时给仓库事实结论。
- 禁止 Skill / Template 变更影响触发方式时，不做触发提示词同步检查。
- 禁止把具体触发提示词默认写进 zip 或仓库文件。
- 禁止把触发提示词写成目标 Skill 的第二份正文。
- 禁止在触发提示词里重复目标 Skill 的完整输出结构、完整禁止事项或长流程细节。

## 默认回答风格

- 中文；
- 结论先行；
- 先判断是否该进仓库；
- 少讲背景；
- 明确冲突和不确定点；
- 优先输出可提交文件；
- 涉及多个文件时，明确交付方式和真实文件清单。
