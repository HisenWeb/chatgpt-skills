# 10-prototype-freeze-business-brief — 原型冻结与业务说明交付

## 触发场景

当低保真页面板或可点击原型已经基本确认，用户希望停止原型阶段的反复修改，并整理一份干净业务说明和最终原型入口给后续实现者理解时，启用本 Skill。

典型触发：

- 用户说“原型可以冻结了”“整理给后续实现者看”“生成业务交接文档”“把原型说明清楚”；
- 已经有页面板原型或可点击原型，需要把页面含义、用户流程、关键状态和业务规则整理成一份干净文档；
- 用户不想把旧聊天、截图、日志、diff、草稿等噪音转交给后续实现者；
- 原型阶段已经有多个材料来源，需要压缩成业务和原型两类最终输入；
- 后续实现者只需要理解业务和原型，不需要网页端替它制定开发方案。

## 不触发场景

以下情况不要启用本 Skill：

- 业务范围还没有说明清楚，应先进入 `skills/08-mvp-scope-confirmation.md`；
- 页面结构、状态归属、主流程或点击关系还没确认，应先进入 `skills/09-html-prototype-generator.md`；
- 用户只是继续修原型、补页面、改样式或调整跳转；
- 用户要求写代码、制定技术方案、生成开发计划、拆组件、设计 API 或指定项目文件；
- 用户要求整理完整证据链、日志、diff、旧聊天或审计材料。

如果原型仍然存在会影响业务理解的缺口，不要硬生成交付文档；先指出缺口并回到对应原型阶段修正。

## 核心定位

本 Skill 的定位是：把最终原型背后的业务含义解释清楚，并给出最终原型入口。

它只做业务和原型交付，不做开发指挥。

核心交付是：

```text
prototype-business-brief.md
最终原型入口
```

这份文档面向后续实现者阅读，但不替后续实现者分析代码、选择技术方案、制定开发计划或决定实现方式。

## 工作原则

### 1. 只交干净文档和最终原型

后续实现者不需要一堆历史材料清单。

本 Skill 应把原型阶段产生的业务结论整理成一份干净业务文档，并只附最终原型入口。

不要把以下内容原样转交：

- 全量旧聊天；
- 多版草稿；
- 临时截图清单；
- 调试日志；
- diff；
- 旧方案争论；
- Skill 维护过程；
- “可能有用”的参考材料。

这些材料可以在本阶段用于判断，但不应成为最终业务交付物的一部分。

### 2. 聚焦业务细节和原型细节

文档必须详细说明业务和原型，而不是只给一句概述。

需要说明：

- 原型表达的业务场景；
- 每个页面的业务用途；
- 页面上有哪些关键内容；
- 页面状态归属于哪个页面；
- 用户从开始到结束如何走；
- 用户点击、选择、提交后业务上发生什么；
- 页面里的业务术语、数据和记录含义；
- 哪些问题仍会影响业务理解。

### 3. 不替后续实现者做决定

不要输出技术栈、项目路径、启动命令、测试命令、组件拆分、目标文件、路由实现、API 方案、数据库设计、开发计划、验收命令或执行端规则。

这些由后续实现者基于真实项目自行分析。

本 Skill 只提供业务和原型事实，让后续实现者理解“要实现的业务是什么、原型表达了什么”。

### 4. 不使用总控式范围口吻

不要在最终业务说明里使用“必做 / 不做 / 候选”这类命令式范围清单。

更合适的表达是：

- 这个原型表达了什么；
- 这个原型没有表达什么；
- 页面中哪些内容代表业务流程的一部分；
- 哪些事项仍需业务确认。

最终文档不应看起来像网页端给后续实现者下开发命令。

### 5. 最终原型入口必须明确

如果同时存在页面板和可点击原型，应说明二者用途：

- 页面板原型：用于理解页面结构、信息密度和状态归属；
- 可点击原型：用于理解用户流程、点击关系和交互闭环。

如果只有一种原型，只说明这一种原型的作用。

不要把旧版本、参考截图或中间原型混入最终入口。

### 6. 页面状态归属必须清楚

弹窗、空状态、筛选展开态、提交确认态、加载态、完成态等，必须说明归属于哪个页面。

不要把状态写成独立页面，除非原型本身明确就是独立页面。

页面状态说明应使用“页面名 + 状态名”的语义，例如：

```text
测试结果页：有错题状态
答题记录页：无记录空状态
题纲列表页：筛选展开态
```

### 7. 只保留会影响业务理解的未决问题

未决问题不是 TodoList，也不是开发任务。

只列会影响后续理解业务或原型的问题，例如：

- 页面语义不清；
- 某个状态代表的业务情况不清；
- 关键交互后的业务结果不清；
- 页面数据含义不清。

不要列技术实现问题、项目结构问题、执行端流程问题或非阻塞视觉偏好。

## 输出要求

### 1. 默认输出 prototype-business-brief.md

推荐结构：

```text
# Prototype Business Brief

## 1. 原型说明
- 这个原型表达的业务场景是什么。

## 2. 最终原型入口
- 页面板原型：...
- 可点击原型：...
- 两者分别用于理解什么。

## 3. 页面说明
### 页面 A
- 页面用途：...
- 页面关键内容：...
- 页面表达的业务含义：...

### 页面 B
- ...

## 4. 用户流程
- 从用户视角描述从开始到结束怎么走。
- 只写业务流程，不写技术路由。

## 5. 页面状态
- 页面 A：状态名 — 代表什么业务情况。
- 页面 B：状态名 — 代表什么业务情况。

## 6. 关键交互
- 用户点击 / 选择 / 提交后，业务上发生什么。
- 返回、取消、重测、继续等交互代表什么。

## 7. 术语与数据含义
- 术语：业务含义。
- 数据 / 记录：页面中代表什么。

## 8. 仍需业务确认的问题
- 只列影响理解原型的问题。
```

### 2. 输出必须给出交付状态

结尾给出：

- `READY_FOR_BUSINESS_HANDOFF`：业务说明和最终原型入口足够清楚，可以交给后续实现者理解；
- `NEEDS_PROTOTYPE_CLARIFICATION`：原型仍有影响理解的问题，需要回到页面板或可点击原型阶段修正；
- `NEEDS_BUSINESS_CLARIFICATION`：业务说明仍不清楚，需要回到 MVP 业务范围说明；
- `NOT_A_FREEZE_TASK`：当前不是原型冻结后的业务说明交付任务。

### 3. 文件交付规则

如果用户要求文件或内容较长，应生成：

```text
prototype-business-brief.md
```

如果最终原型是本轮生成的实际 HTML 文件，应同时提供原型入口链接。

如果原型文件并非当前窗口可访问，不能假装已经迁移；应明确需要用户提供最终原型文件或入口。

### 4. 与其他 Skill 的衔接

- 业务说明不清：回到 `skills/08-mvp-scope-confirmation.md`；
- 页面结构、状态归属、用户流程或点击关系不清：回到 `skills/09-html-prototype-generator.md`。

不要在本 Skill 中继续扩展原型，也不要输出开发计划。

## 禁止事项

- 禁止替后续实现者制定开发计划、技术方案或执行规则。
- 禁止指定项目路径、文件、组件、路由实现、API 或数据库结构。
- 禁止绑定任何具体执行工具、规格框架或 agent 工作流。
- 禁止把旧聊天、截图、日志、diff、旧原型或过程证据作为材料清单转交。
- 禁止使用“必做 / 不做 / 候选”这类总控式开发口吻。
- 禁止把未确认的业务假设写成事实。
- 禁止把状态误写成独立页面。
- 禁止把页面板和可点击原型的中间版本混作最终入口。
- 禁止输出技术验收命令、测试命令或代码修改建议。
- 禁止把本 Skill 变成开发提示词生成器、规格框架输入生成器或执行端工作流启动器。
