# 09-html-prototype-generator — HTML 原型生成

> 目的：把已收敛的需求、页面规格或交接材料，转成可本地预览、可审图、可继续修改的 HTML/CSS/JS 低保真原型。  
> 定位：生成实际 HTML 原型，不生成图片参考图，不回到设计工具流程。  
> 核心交付：输出实际原型文件，并给出可直接打开的预览入口；默认不生成压缩包，除非用户明确要求，或多文件本地交付确实需要。  
> 默认语言：中文说明；代码、文件名、路径、选择器、命令保持英文。

---

## 核心定位

本 Skill 负责把已确认业务材料转成可预览、可审图、可修改的静态 HTML 原型，并作为后续冻结评审的原型事实源。

用户确认最终原型后，才生成：

```text
prototype-delivery.md
prototype/
```

- `prototype-delivery.md`：说明最终原型实际表达了什么；
- `prototype/`：放最终确认版本的原型文件和必要资源。

本 Skill 不重新定义业务范围；业务范围以 `mvp-business-scope.md` 或用户等价确认材料为准。若最终原型与业务范围源不一致，只在 `prototype-delivery.md` 的“原型未覆盖 / 待确认”中标出，不自行改写业务范围。

---

## 触发场景

用户已有相对收敛的业务范围、页面规格、交接材料、scope、brief、PRD 或既有 HTML 原型，并希望生成或修正可本地预览的 HTML/CSS/JS 原型。

典型任务：

- 根据 `mvp-business-scope.md` 或等价业务范围说明生成 HTML 原型；
- 生成页面板，用于审页面结构、信息密度和状态归属；
- 生成可点击原型，用于验证用户流程、路由关系和交互闭环；
- 根据 handoff / scope / brief / PRD 生成可预览原型；
- 根据已有 HTML 原型做最小改动修正；
- 用户确认最终原型后，生成 `prototype-delivery.md` 和 `prototype/`。

真实产品开发前先做低保真 HTML 原型，本 Skill 仍适用。

## 不触发场景

- 业务场景、用户流程、页面含义或关键状态未收敛：先进入 `skills/08-mvp-scope-confirmation.md`；
- 用户只是要图片参考图、高保真设计稿或设计工具产物；
- 用户只是评审 Prompt、Skill、Template、文档或仓库结构；
- 用户要求生产级前端工程、后端系统、技术方案、开发计划、API 或数据库设计；
- 已有 `mvp-business-scope.md`、`prototype-delivery.md` 和最终 `prototype/`，只做冻结评审与最终打包：进入 `skills/10-prototype-freeze-gate.md`；
- 当前任务是 Skill 仓库维护：优先启用 `skills/00-skill-authoring.md`。

## 工作原则

启用后先判断模式，不要直接写代码或生成最终交付。

### 执行模式

| 模式 | 触发条件 | 本轮动作 |
|---|---|---|
| A. 设置确认 | 目标端、输出形态、预览目标、路由实现不明确 | 给建议设置、待确认项、必要路由草案 |
| B. 页面板原型 | 用户要页面总览、状态板、参考板、审结构 / 密度 | 生成页面板 HTML 与预览入口 |
| C. 可点击原型 | 用户要可点击、跳转、跑通流程、交互闭环 | 先确认路由关系和实现模式，再生成 |
| D. 既有原型修正 | 用户上传旧 HTML、截图、报错或具体反馈 | 定位问题，最小改动修正 |
| E. 最终交付 | 用户明确确认最终原型 | 生成 `prototype-delivery.md` 和最终 `prototype/` |

硬性流程：

```text
先确认原型设置
→ 生成页面板 / 可点击原型
→ 用户评审并明确确认最终原型
→ 才生成 prototype-delivery.md + prototype/
```

未确认最终原型前，只能输出设置确认、路由表草案、可评审原型、预览入口、问题清单或修改建议；不能固化为 `prototype-delivery.md` 和最终 `prototype/`。

“继续”“先这样”“看起来还行”“可以先用”等模糊反馈不等于最终确认；必须出现“确认最终原型 / 可以冻结 / 生成 prototype-delivery / 生成最终交付”这类明确语义，才进入最终交付模式。

## 生成前判断

最低判断项：

- 本轮目标：全量默认态、核心路径、指定页面、状态补充，还是修正既有原型；
- 目标端：手机端、小程序、App、H5、PC Web、管理台、官网或其他载体；
- 输出形态：页面板模式，还是可点击原型模式；
- 预览目标：ChatGPT 内嵌预览、本地浏览器打开，还是两者都要兼容；
- 可点击原型的路由实现：单 HTML 内部视图切换，还是多 HTML 相对路径跳转；
- 页面范围：要生成哪些页面或哪条主路径；
- 业务边界来源：来自 `mvp-business-scope.md` 或用户等价确认材料；09 只识别原型生成边界，不重新定义业务范围；
- 冲突：需求、交接材料、截图或旧原型是否互相矛盾。

判断规则：

- 明确“小程序 / App / H5 / 移动端”可建议手机端；明确“PC Web / 管理台 / 官网 / 桌面端”可建议 PC 端；目标端不明确则问或给建议设置；
- 明确“页面总览 / 状态板 / 参考板 / 先看所有页面效果 / 多个页面并排”时，建议页面板模式；
- 明确“可点击 / 跑通流程 / 跳转 / 走完整路径 / 交互闭环”时，建议可点击原型模式；
- 输出形态或预览目标不明确时，必须先询问或输出确认卡；
- 可点击原型必须确认路由表和路由实现模式；用户未提供时，先给建议路由表和实现模式让用户确认；
- 页面范围不明确时，先问；
- 视觉要求缺失但页面范围、目标端、输出形态明确时，可按低保真默认风格做；
- 材料冲突影响页面结构、输出形态或原型生成边界时，先指出冲突，不要自行合并生成。

确认卡推荐：

```text
请先确认本轮原型设置：
1. 目标端：手机端 / PC 端 / 两端都要 / 其他
2. 输出形态：页面板模式 / 可点击原型模式
3. 预览目标：ChatGPT 内嵌预览 / 本地浏览器打开 / 两者都要兼容
4. 路由实现：单 HTML 内部视图切换 / 多 HTML 相对路径跳转 / 页面板 Tab 切换 / 由材料和预览目标决定
我根据材料的建议是：...
```

材料可明确推断时，给“建议设置 + 待确认项”；存在关键歧义时，停止并询问。

## 输入材料识别

不要只按文件名判断材料用途，必须结合内容判断。

- 需求 / 范围材料：页面清单、业务边界来源、术语、已确认不做项、验收标准；09 只识别原型生成边界，不重写业务范围；
- 页面 / 交互规格：页面模块、状态、按钮、跳转关系；
- 当前状态 / 交接材料：本轮先做什么、旧方案废弃点、最新结论；
- 视觉参考：截图、旧设计、旧原型，只用于风格、布局密度或问题定位，不自动覆盖需求；
- 实现证据：旧 HTML、日志、测试结果、diff，用于定位或复用结构，不自动视为最新需求源。

材料冲突时：先指出冲突点和影响范围（页面范围、交互状态、视觉风格或实现细节），等用户确认后再生成或修改。

## 生成策略

按材料、目标和已确认输出形态选择策略，不内置固定项目页面名或固定文件结构。

| 策略 | 适合场景 | 边界 |
|---|---|---|
| 全页面默认态 | 统一整体视觉、信息密度、页面结构 | 每页只做默认态，不补复杂状态 |
| 核心路径原型 | 验证主流程能否走通 | 只生成主路径必需页面和最少交互 |
| 指定页面原型 | 用户只要求某页、弹窗或局部模块 | 不扩展到未要求页面 |
| 状态补充 | 已有默认态后补空/错/完成/筛选等状态 | 状态数量和含义来自材料或确认，不自由扩展 |
| 既有原型修正 | 用户上传旧 HTML、截图或反馈 | 最小改动，不整包重写 |

未指定策略时：页面范围完整可做全页面默认态；主流程重要可做核心路径；只提供单页规格只做指定页面；已有原型并有反馈则按反馈修正。

输出形态关系：

- 全页面默认态、状态补充、页面总览通常适合页面板模式；
- 核心路径、按钮流转、交互闭环通常适合可点击原型模式，且必须先确认路由；
- 指定页面可做页面板单屏或独立可点击页面；
- 既有原型修正沿用原输出形态，除非用户明确要求切换。

## 页面板模式

页面板用于快速审图、对比页面结构、确认信息密度和状态归属。它不是滚动目录，也不是把所有状态提升成顶层页面。

规则：

- 顶层导航默认对应真实页面或一级页面组，不对应零散按钮、局部模块或实现文件；
- 页面多时默认用顶部 Tab / 分组按钮切换当前可见页面板；除非用户明确要求“纵向总览 / 全部平铺 / 一屏审全部”，不要把锚点滚动作为主要导航；
- 顶部按钮 / Tab 应直接切换可见页面板，不应只是滚动到锚点；
- 弹窗、空状态、筛选展开态、提交确认态、加载态、完成态等状态，应归入所属页面内部；
- 只有用户明确要求“状态板 / 状态总览”时，才拆成独立状态卡片或独立分组；
- 同一顶层页面下可展示多个状态卡片，标题使用“页面名 + 状态名”，避免状态被误判为独立页面；
- 材料未说明页面状态时，只补默认态和核心链路必要状态，不自由扩展大量状态。

---

## 可点击原型模式

可点击原型用于验证核心路径、按钮流转和交互闭环。生成前必须确认页面跳转关系和路由实现模式。

### 路由确认

路由确认不是正式前端路由配置，只是低保真原型的跳转契约。

需要说明：原型入口、页面跳转关系、关键按钮去向、URL 参数或轻量状态参数、条件跳转（继续、重测、返回、取消、提交等）。

处理规则：

- 用户已提供路由表、流程图或 Mermaid 时，以用户材料为准；
- 用户未提供，但页面范围和主流程明确时，先输出轻量路由表让用户确认；
- 路由复杂时可附 Mermaid 辅助审核；Mermaid 只作审核材料，不作 HTML 运行依赖；
- 页面板模式通常不需要路由表，除非用户要求补充说明。

路由表字段：

```text
| 当前页面 | 触发入口 | 目标页面 / 目标视图 | 参数 / 状态 | 说明 |
|---|---|---|---|---|
```

### 路由实现模式

1. 单 HTML 内部视图切换
   - 适合 ChatGPT 内嵌预览，或希望一个入口兼容预览和本地打开；
   - 使用 `data-route`、内部状态或轻量 hash 表达当前视图；
   - 点击主要在当前 HTML 内切换视图，不依赖跨文件跳转。

2. 多 HTML 相对路径跳转
   - 适合下载到本地浏览器打开，或交给执行端继续拆分；
   - 页面间使用相对路径和查询参数表达跳转；
   - 交付前确认入口页和主要链路可达。

3. 页面板 Tab 切换
   - 适合页面板模式；
   - 顶部 Tab / 分组按钮直接切换页面板；
   - 不把锚点滚动作为主要导航，除非用户明确要求滚动总览。

判断：预览目标包含 ChatGPT 内嵌预览时，优先建议单 HTML 内部视图切换；用户明确要求本地目录、多页面文件或后续交给执行端开发时，可建议多 HTML 相对路径跳转；用户主要审结构、密度和状态归属时，建议页面板 Tab 切换；不确定时让用户选，不自行默认。

## 预览容器基线

手机端和 PC 端都应有明确预览容器，不让内容直接铺满浏览器窗口。

推荐结构：

```text
preview-stage
└── prototype-container
    └── prototype-content
```

规则：

- `preview-stage`：预览舞台，默认水平和垂直居中；
- `prototype-container`：原型边界，根据目标端选择手机容器、桌面应用容器、浏览器窗口或页面画布；
- `prototype-content`：页面内容区，长内容在容器内部滚动，不撑高外层舞台；
- 手机端 / 小程序 / App / H5 使用手机尺寸容器；PC Web / 管理台 / 官网使用桌面或页面容器；
- 底部导航、顶部栏、侧边栏、底部操作区等固定区域固定在容器内，而不是固定到整个浏览器窗口。

## 文件结构与交付

文件结构由材料、页面范围和生成策略决定，不写死项目路径或页面文件名。

原则：

- 入口文件能直接打开，预览入口来自实际原型目录；
- 单入口预览优先单文件 HTML 内联 CSS/JS，避免预览环境无法访问相对资源；
- 多页面原型用相对路径跳转；CSS / JS 使用相对路径；
- 页面文件名根据页面语义生成；共享样式/脚本可集中放置，但目录名不固定；
- 用户材料已有命名约定时，以用户材料为准；
- 拆分 CSS / JS / 数据文件时，确保入口文件能通过相对路径访问，并在验收输出中说明检查结果；
- 默认不生成 zip；用户明确要求打包，或多文件目标是下载到本地使用时，可附加 zip；
- 不在聊天里粘贴完整 HTML/CSS/JS 大段源码，除非用户明确要求。

可选建议：

- ChatGPT 内嵌预览或两者兼容：单 HTML、内联 CSS/JS、内部视图切换；
- 单入口页面板：一个 HTML 文件内联少量 CSS / JS；
- 本地浏览器优先的多页面可点击原型：页面 HTML + 共享样式文件 + 共享脚本文件；
- 数据较多且预览环境支持相对资源时，可单独放示例数据文件；
- 无法确认预览环境能否访问同目录资源时，优先单文件保证可审图。

## 实现原则

- 低保真优先：目标是可审图、可跑通路径、可快速迭代；不要追求高保真、复杂动画、精美插画、生产级架构、过度组件化或构建工具链；
- 本地静态可运行：默认不依赖后端、数据库、登录、云同步、安装依赖、构建命令或外部 CDN；优先原生 HTML/CSS/JS；除非用户明确要求，不引入框架；
- 数据集中：示例数据应集中管理，避免多处复制；页面共用同一份示例数据或清晰常量；术语一致；示例文案接近真实使用场景，不只写“标题 1 / 内容 A”；
- 简单跳转：可使用多 HTML 文件、URL 参数或单文件内部视图表达跳转；跳转关系清楚，参数命名可读，不引入复杂路由器，不让不同入口指向矛盾数据结构；ChatGPT 内嵌预览优先时，主要点击尽量在当前容器内切换视图。

---

## 状态边界

允许：示例数据、少量本地临时状态、简单按钮状态、弹窗显示 / 隐藏、页面跳转、查询参数读取、简单筛选、简单搜索过滤。

默认用内存、查询参数或示例数据表达状态。只有确实需要跨页面保留少量状态时，才使用本地存储，并说明用途；不要写入大体积数据。

禁止默认引入：真实登录、后端接口、云同步、真实业务持久化、在线代码运行、未经需求确认的扩展功能。

---

## 预览自检

预览是实际原型入口，不是额外参考图。生成后尽量自检：

- 入口文件可直接打开；
- 首屏有可见内容，不是只有空容器等待脚本渲染；
- CSS / JS 路径相对可用，或已采用单文件内联；
- 多页面原型能从入口页进入主要页面；单文件原型主要视图切换可用；
- 跳转方式符合已确认路由实现模式；
- 页面板顶部 Tab / 分组按钮直接切换页面板，不误触发锚点滚动，除非用户明确要求滚动总览；
- 页面状态出现在所属页面或所属分组下，不默认提升为独立顶层页面；
- 关键按钮和跳转不指向不存在的文件或视图；
- 原型容器尺寸受控，长内容在容器内部滚动；
- 页面不能明显空白或报错。

如已实际检查文件，说明检查结果；未检查的项目标注“未检查”，不要假装通过。

## 最终原型交付要求

只有用户明确确认最终原型后，才生成：

```text
prototype-delivery.md
prototype/
```

`prototype/` 只放最终确认版本的原型文件和必要资源，不放旧版本、草稿、截图、日志或过程材料。

`prototype/` 固定入口命名：

```text
prototype/
├─ index.html        # 可点击原型或单入口原型，如有
├─ page-board.html   # 页面板原型，如有
└─ assets/           # 必要资源，如有
```

规则：如果只有一个原型，统一使用 `prototype/index.html`；如果同时有页面板和可点击原型，页面板使用 `prototype/page-board.html`，可点击或主入口使用 `prototype/index.html`；`assets/` 只放最终原型必要资源。

## 修改既有原型

当用户要求修 Bug 或改细节：

1. 先定位问题页面、选择器、数据入口或跳转入口；
2. 列出真实修改点；
3. 只改必要文件；
4. 不顺手重构；
5. 不引入框架；
6. 不把临时验证直接正式化；
7. 不因局部问题整包重写，除非用户明确要求重做。

修改后说明：改了哪些文件、解决了什么问题、还有哪些未处理点。

## 禁止事项

- 禁止在业务范围不清时直接生成原型；
- 禁止在目标端、输出形态、预览目标或可点击路由实现不明确时直接生成；
- 禁止在用户确认最终原型前生成 `prototype-delivery.md` 或最终 `prototype/`；
- 禁止用原型交付说明重新定义业务范围；
- 禁止把旧聊天、截图、日志、diff、旧版本原型或过程材料混入最终 `prototype/`；
- 禁止生成图片参考图替代实际 HTML 原型；
- 禁止默认引入框架、后端、数据库、登录、云同步、构建工具链或外部 CDN；
- 禁止把页面状态默认提升为独立顶层页面；
- 禁止页面板主导航默认使用锚点滚动；
- 禁止让关键按钮或跳转指向不存在的文件或视图；
- 禁止因为局部问题整包重写已有原型，除非用户明确要求重做；
- 禁止在聊天里粘贴完整大段源码，除非用户明确要求。
