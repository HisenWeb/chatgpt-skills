# 08-html-prototype-generator — HTML 低保真原型生成

> 目的：把已收敛的需求、页面规格或交接材料，转成可本地预览、可审图、可继续修改的 HTML/CSS/JS 低保真原型。  
> 定位：生成实际 HTML 原型，不生成图片参考图，不回到设计工具流程。  
> 核心交付：只输出实际原型文件，并给出可直接打开的预览入口；不生成压缩包。  
> 默认语言：中文说明；代码、文件名、路径、选择器、命令保持英文。

---

## 1. 适用场景

当用户已经有相对收敛的材料，并希望直接得到可预览 HTML 原型时，启用本 Skill。

典型请求：

- 根据需求生成低保真原型；
- 根据页面规格生成 HTML 页面；
- 根据 handoff / scope / brief / PRD 生成可预览原型；
- 在已有 HTML 原型上按反馈修改页面。

不适用：

- 只是讨论产品方向，页面范围还未收敛；
- 只是要图片参考图；
- 只是评审 Prompt / Skill / 文档；
- 要直接开发生产级前端工程、组件库或后端系统。

如果用户是在真实产品开发前先做低保真 HTML 原型，本 Skill 仍适用。

---

## 2. 执行前判断

生成前先判断材料是否足够，不要马上写代码。

最低判断项：

- 本轮目标是否明确：全量默认态、核心路径、指定页面，还是基于已有原型修正；
- 页面范围是否明确：至少知道要生成哪些页面或哪条主路径；
- 边界是否明确：哪些功能不做、哪些只是占位；
- 是否存在明显冲突：例如需求、交接材料、截图或旧原型互相矛盾。

如果信息不足：

- 缺少页面范围时，先问；
- 缺少视觉要求但页面范围明确时，可以按低保真默认风格先做；
- 材料冲突会影响页面结构或功能边界时，先指出冲突，不要自行合并后生成。

---

## 3. 输入材料识别

不要只按文件名判断材料用途。文件名只是线索，必须结合内容判断。

常见材料角色：

- 需求 / 范围材料：确定页面清单、功能边界、术语、不做项和验收标准。
- 页面 / 交互规格：确定每个页面要包含的模块、状态、按钮和跳转关系。
- 当前状态 / 交接材料：确定本轮先做什么、哪些旧方案已废弃、哪些结论是最新的。
- 视觉参考：截图、旧设计、旧原型，仅用于风格、布局密度或问题定位，不自动覆盖需求。
- 实现证据：旧 HTML、日志、测试结果、diff，用于定位问题或复用结构，不自动视为最新需求源。

如果材料之间冲突：

- 先指出冲突点；
- 说明影响范围，例如页面范围、交互状态、视觉风格或实现细节；
- 等用户确认后再生成或修改原型。

---

## 4. 生成策略

根据用户材料和当前目标选择生成方式，不内置固定项目页面名或固定文件结构。

常见策略：

1. 全页面默认态
   - 适合先统一整体视觉、信息密度和页面结构。
   - 每个页面只做默认状态，不补复杂状态。

2. 核心路径原型
   - 适合先验证主流程能否走通。
   - 只生成主路径必需页面和最少交互。

3. 指定页面原型
   - 适合用户只要求某个页面、弹窗或局部模块。
   - 不扩展到未要求页面。

4. 状态补充
   - 适合已有默认态后，补充空状态、错误状态、完成状态、筛选状态等。
   - 状态数量和含义应来自用户材料或用户确认，不让模型自由扩展太多。

5. 既有原型修正
   - 适合用户上传旧 HTML、截图或反馈。
   - 遵守最小改动，不整包重写。

如果用户没有指定策略：

- 页面范围很完整时，可先做全页面默认态；
- 主流程很重要时，可先做核心路径；
- 只提供单页规格时，只做指定页面；
- 已有原型并有反馈时，按反馈修正。

---

## 5. 交付规则

每次生成原型时，只交付实际原型文件和可直接打开的预览入口。

要求：

- 预览入口必须来自实际原型目录；
- 不生成压缩包；
- 不提示打包；
- 不用图片或参考板替代实际原型；
- 不在聊天里粘贴完整 HTML/CSS/JS 大段源码，除非用户明确要求。

回复格式：

```text
原型预览入口：
- sandbox:/mnt/data/.../<entry-file>.html

本轮覆盖：
- ...

未覆盖 / 下一步：
- ...
```

---

## 6. 文件结构原则

文件结构由用户材料、本轮页面范围和生成策略决定，不写死项目路径或页面文件名。

原则：

- 入口文件应能直接打开；
- 多页面原型使用相对路径跳转；
- CSS / JS 使用相对路径；
- 页面文件名应根据页面语义生成；
- 共享样式和共享脚本可以集中放置，但目录名不固定；
- 若用户材料已有命名约定，以用户材料为准。

可选推荐：

- 小型单页原型：一个 HTML 文件内联少量 CSS / JS；
- 多页面低保真原型：页面 HTML + 一个共享样式文件 + 一个共享脚本文件；
- 数据较多时：单独放一个示例数据文件，避免多页面重复写死数据。

不要把某个项目的页面名、目录名或数据文件名写成通用规则。

---

## 7. 实现原则

### 7.1 低保真优先

目标是可审图、可跑通路径、可快速迭代。

不要追求：

- 高保真设计稿；
- 复杂动画；
- 精美插画；
- 生产级工程架构；
- 过度组件化；
- 构建工具链。

### 7.2 本地静态可运行

默认不依赖：

- 后端；
- 数据库；
- 登录；
- 云同步；
- 安装依赖；
- 构建命令；
- 外部 CDN。

优先使用原生 HTML/CSS/JS。除非用户明确要求，不引入框架。

### 7.3 数据组织

示例数据应集中管理，避免多处复制导致不一致。

要求：

- 页面共用同一份示例数据或同一组清晰常量；
- 保持术语一致；
- 示例文案要接近真实使用场景，不要只写“标题 1 / 内容 A”；
- 若用户材料已有数据结构或命名约定，以用户材料为准。

### 7.4 简单跳转

低保真原型可以使用多 HTML 文件和 URL 参数表达跳转关系。

要求：

- 跳转关系清楚；
- 参数命名保持可读；
- 不为了原型引入复杂路由器；
- 不让不同入口指向互相矛盾的数据结构。

---

## 8. 状态处理边界

允许：

- 示例数据；
- 少量本地临时状态；
- 简单按钮状态；
- 弹窗显示 / 隐藏；
- 页面跳转；
- 查询参数读取；
- 简单筛选；
- 简单搜索过滤。

状态默认以内存、查询参数或示例数据表达。

只有确实需要跨页面保留少量状态时，才使用本地存储，并说明用途；不要写入大体积数据。

禁止默认引入：

- 真实登录；
- 后端接口；
- 云同步；
- 真实业务持久化；
- 在线代码运行；
- 未经需求确认的扩展功能。

---

## 9. 预览要求

预览不是额外参考图，而是实际原型入口。

必须满足：

- 入口文件可直接打开；
- CSS / JS 路径相对可用；
- 多页面原型能从入口页进入主要页面；
- 关键按钮和跳转不应指向不存在的文件；
- 页面不能明显空白或报错。

---

## 10. 验收输出

生成预览后，回复必须列出：

```text
已生成：
- ...

原型预览入口：
- ...

已覆盖：
- ...

未覆盖 / 下一步：
- ...

注意：
- ...
```

如已实际检查文件，应说明：

- 入口文件存在；
- CSS / JS 路径可用；
- 主要页面链接存在；
- 未引入范围外功能。

---

## 11. 修改规则

当用户要求修 Bug 或改细节：

1. 先定位问题页面、选择器、数据入口或跳转入口；
2. 列出真实修改点；
3. 只改必要文件；
4. 不顺手重构；
5. 不引入框架；
6. 不把临时验证直接正式化；
7. 不因局部问题整包重写，除非用户明确要求重做。

修改后说明：

- 改了哪些文件；
- 解决了什么问题；
- 还有哪些未处理点。

---

## 12. 输出要求

默认输出：

- 一个可直接打开的实际原型预览链接；
- 简短说明；
- 本轮覆盖范围；
- 下一步建议。

不要输出压缩包。  
不要提示打包。  
不要生成图片参考图。  
不要把完整源码大段贴在聊天里，除非用户明确要求。
