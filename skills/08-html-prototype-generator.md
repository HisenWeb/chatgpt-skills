# 08-html-prototype-generator — HTML 低保真原型生成

> 目的：把已收敛的需求、页面规格或交接材料，转成可本地预览、可审图、可继续修改的 HTML/CSS/JS 低保真原型。  
> 定位：生成实际 HTML 原型，不生成图片参考图，不回到设计工具流程。  
> 核心交付：只输出实际原型文件，并给出可直接打开的预览入口；不生成压缩包。生成前必须确认输出形态：页面板模式或可点击原型模式。  
> 默认语言：中文说明；代码、文件名、路径、选择器、命令保持英文。

---

## 1. 适用场景

当用户已经有相对收敛的材料，并希望直接得到可预览 HTML 原型时，启用本 Skill。

典型请求：

- 根据需求生成低保真原型；
- 根据页面规格生成 HTML 页面；
- 根据 handoff / scope / brief / PRD 生成可预览原型；
- 在已有 HTML 原型上按反馈修改页面。

不适用：

- 只是讨论产品方向，页面范围还未收敛；
- 只是要图片参考图；
- 只是评审 Prompt / Skill / 文档；
- 要直接开发生产级前端工程、组件库或后端系统。

如果用户是在真实产品开发前先做低保真 HTML 原型，本 Skill 仍适用。

---

## 2. 执行前判断

生成前先判断材料是否足够，不要马上写代码。

最低判断项：

- 本轮目标是否明确：全量默认态、核心路径、指定页面、状态补充，还是基于已有原型修正；
- 输出形态是否明确：页面板模式，还是可点击原型模式；
- 页面范围是否明确：至少知道要生成哪些页面或哪条主路径；
- 边界是否明确：哪些功能不做、哪些只是占位；
- 是否存在明显冲突：例如需求、交接材料、截图或旧原型互相矛盾。

输出形态说明：

1. 页面板模式
   - 在一个 HTML 中集中展示多个页面 / 页面状态，可用 Tab、分组按钮、分区或并排卡片组织；
   - 适合审整体视觉、页面结构、信息密度、全页面默认态或状态板；
   - 不强调真实跳转；页面较多时默认用 Tab / 分组按钮直接切换当前可见页面板，而不是把锚点滚动作为主要导航。

2. 可点击原型模式
   - 生成实际入口页和页面跳转；
   - 适合验证核心路径、按钮流转和交互闭环；
   - 可以用多 HTML 文件或单 HTML 多视图实现，具体由材料决定。

判断规则：

- 如果用户明确说“页面总览 / 状态板 / 参考板 / 先看所有页面效果 / 多个手机页面并排”，默认采用页面板模式；
- 如果用户明确说“可点击 / 跑通流程 / 跳转 / 走完整路径 / 交互闭环”，默认采用可点击原型模式；
- 可点击原型模式下还需要确认路由表；如果用户未提供，先生成建议路由表让用户确认；
- 如果用户没有明确说明输出形态，必须先询问用户选择哪种模式。

如果信息不足：

- 缺少输出形态时，先问；
- 缺少页面范围时，先问；
- 缺少视觉要求但页面范围和输出形态明确时，可以按低保真默认风格先做；
- 材料冲突会影响页面结构、输出形态或功能边界时，先指出冲突，不要自行合并后生成。

## 3. 输入材料识别

不要只按文件名判断材料用途。文件名只是线索，必须结合内容判断。

常见材料角色：

- 需求 / 范围材料：确定页面清单、功能边界、术语、不做项和验收标准。
- 页面 / 交互规格：确定每个页面要包含的模块、状态、按钮和跳转关系。
- 当前状态 / 交接材料：确定本轮先做什么、哪些旧方案已废弃、哪些结论是最新的。
- 视觉参考：截图、旧设计、旧原型，仅用于风格、布局密度或问题定位，不自动覆盖需求。
- 实现证据：旧 HTML、日志、测试结果、diff，用于定位问题或复用结构，不自动视为最新需求源。

如果材料之间冲突：

- 先指出冲突点；
- 说明影响范围，例如页面范围、交互状态、视觉风格或实现细节；
- 等用户确认后再生成或修改原型。

---

## 4. 生成策略

根据用户材料、当前目标和已确认的输出形态选择生成方式，不内置固定项目页面名或固定文件结构。

常见策略：

1. 全页面默认态
   - 适合先统一整体视觉、信息密度和页面结构。
   - 每个页面只做默认状态，不补复杂状态。

2. 核心路径原型
   - 适合先验证主流程能否走通。
   - 只生成主路径必需页面和最少交互。

3. 指定页面原型
   - 适合用户只要求某个页面、弹窗或局部模块。
   - 不扩展到未要求页面。

4. 状态补充
   - 适合已有默认态后，补充空状态、错误状态、完成状态、筛选状态等。
   - 状态数量和含义应来自用户材料或用户确认，不让模型自由扩展太多。

5. 既有原型修正
   - 适合用户上传旧 HTML、截图或反馈。
   - 遵守最小改动，不整包重写。

如果用户没有指定策略：

- 页面范围很完整时，可先做全页面默认态；
- 主流程很重要时，可先做核心路径；
- 只提供单页规格时，只做指定页面；
- 已有原型并有反馈时，按反馈修正。

输出形态与策略的关系：

- 全页面默认态、状态补充、页面总览，通常更适合页面板模式；
- 核心路径、按钮流转、交互闭环，通常更适合可点击原型模式；可点击原型模式生成前应先完成路由确认；
- 指定页面原型可根据用户目标选择页面板中的单屏，或独立可点击页面；
- 既有原型修正沿用原原型的输出形态，除非用户明确要求切换。

### 4.1 页面板组织规则

页面板模式不是滚动目录，也不是把所有状态提升成顶层页面。默认组织方式应便于快速审图、对比页面结构和定位页面状态。

规则：

- 顶层导航默认对应真实页面或一级页面组，不对应零散按钮、局部模块或实现文件。
- 页面数量较多时，默认使用顶部 Tab / 分组按钮切换当前可见页面板；除非用户明确要求“纵向总览 / 全部平铺 / 一屏审全部”，不要把锚点滚动作为主要导航。
- 弹窗、空状态、筛选展开态、提交确认态、加载态、完成态等页面状态，应归入所属页面内部展示；只有用户明确要求做“状态板 / 状态总览”时，才拆成独立状态卡片或独立分组。
- 同一顶层页面下可以展示多个状态卡片，标题应使用“页面名 + 状态名”的语义，避免把状态误判为独立页面。
- 如果材料没有说明页面状态，只补默认态和核心链路必要状态，不自由扩展大量状态。

## 5. 可点击原型模式的路由确认

如果输出形态选择“可点击原型模式”，生成前应确认页面跳转关系。

路由确认用于说明：

- 原型入口；
- 页面之间的跳转关系；
- 关键按钮点击后去哪里；
- URL 参数或轻量状态参数；
- 条件跳转，例如继续、重测、返回、取消、提交等。

路由确认不是正式前端路由配置，只是低保真原型的跳转契约。

处理规则：

- 如果用户已提供路由表、流程图或 Mermaid，以用户材料为准；
- 如果用户未提供，但页面范围和主流程明确，先输出一版轻量路由表让用户确认；
- 路由关系较复杂时，可以同时给出 Mermaid 流程图辅助审核；
- Mermaid 只作为路由审核材料，不作为 HTML 原型运行依赖；
- 页面板模式不需要路由表，除非用户明确要求补充说明。

路由表推荐字段：

```text
| 当前页面 | 触发入口 | 目标页面 / 目标视图 | 参数 / 状态 | 说明 |
|---|---|---|---|---|
```

## 6. 交付规则

每次生成原型时，只交付已确认输出形态下的实际原型文件和可直接打开的预览入口。

要求：

- 预览入口必须来自实际原型目录；
- 不生成压缩包；
- 不提示打包；
- 不用图片或参考板替代实际原型；
- 如果只给一个可直接打开的预览入口且不生成压缩包，优先生成单文件 HTML 内联 CSS/JS，避免预览环境无法访问相对资源；
- 如果拆分 CSS / JS / 数据文件，必须确保入口文件能通过相对路径访问这些资源，并在验收输出中说明检查结果；
- 不在聊天里粘贴完整 HTML/CSS/JS 大段源码，除非用户明确要求。

回复格式：

```text
原型预览入口：
- sandbox:/mnt/data/.../<entry-file>.html

本轮覆盖：
- ...

未覆盖 / 下一步：
- ...
```

## 7. 文件结构原则

文件结构由用户材料、本轮页面范围和生成策略决定，不写死项目路径或页面文件名。

原则：

- 入口文件应能直接打开；
- 多页面原型使用相对路径跳转；
- CSS / JS 使用相对路径；
- 页面文件名应根据页面语义生成；
- 共享样式和共享脚本可以集中放置，但目录名不固定；
- 若用户材料已有命名约定，以用户材料为准。

可选推荐：

- 单入口页面板：优先一个 HTML 文件内联少量 CSS / JS，保证预览入口直接可用；
- 多页面可点击原型：页面 HTML + 一个共享样式文件 + 一个共享脚本文件；
- 数据较多且预览环境支持相对资源时：可单独放一个示例数据文件，避免多页面重复写死数据；
- 如果无法确认预览环境是否能访问同目录资源，宁可先用单文件原型保证可审图。

不要把某个项目的页面名、目录名或数据文件名写成通用规则。

## 8. 实现原则

### 8.1 低保真优先

目标是可审图、可跑通路径、可快速迭代。

不要追求：

- 高保真设计稿；
- 复杂动画；
- 精美插画；
- 生产级工程架构；
- 过度组件化；
- 构建工具链。

### 8.2 本地静态可运行

默认不依赖：

- 后端；
- 数据库；
- 登录；
- 云同步；
- 安装依赖；
- 构建命令；
- 外部 CDN。

优先使用原生 HTML/CSS/JS。除非用户明确要求，不引入框架。

### 8.3 数据组织

示例数据应集中管理，避免多处复制导致不一致。

要求：

- 页面共用同一份示例数据或同一组清晰常量；
- 保持术语一致；
- 示例文案要接近真实使用场景，不要只写“标题 1 / 内容 A”；
- 若用户材料已有数据结构或命名约定，以用户材料为准。

### 8.4 简单跳转

低保真原型可以使用多 HTML 文件和 URL 参数表达跳转关系。

要求：

- 跳转关系清楚；
- 参数命名保持可读；
- 不为了原型引入复杂路由器；
- 不让不同入口指向互相矛盾的数据结构。

---

## 9. 状态处理边界

允许：

- 示例数据；
- 少量本地临时状态；
- 简单按钮状态；
- 弹窗显示 / 隐藏；
- 页面跳转；
- 查询参数读取；
- 简单筛选；
- 简单搜索过滤。

状态默认以内存、查询参数或示例数据表达。

只有确实需要跨页面保留少量状态时，才使用本地存储，并说明用途；不要写入大体积数据。

禁止默认引入：

- 真实登录；
- 后端接口；
- 云同步；
- 真实业务持久化；
- 在线代码运行；
- 未经需求确认的扩展功能。

---

## 10. 预览要求

预览不是额外参考图，而是实际原型入口。

必须满足：

- 入口文件可直接打开；
- CSS / JS 路径相对可用；若采用单文件内联，应确认无需外部 CSS / JS；
- 多页面原型能从入口页进入主要页面；
- 页面板模式下，顶部 Tab / 分组按钮应直接切换页面板，不应误触发锚点滚动，除非用户明确要求滚动总览；
- 页面状态应出现在所属页面或所属分组下，不应默认提升为独立顶层页面；
- 关键按钮和跳转不应指向不存在的文件；
- 页面不能明显空白或报错。

---

## 11. 验收输出

生成预览后，回复必须列出：

```text
已生成：
- ...

原型预览入口：
- ...

已覆盖：
- ...

未覆盖 / 下一步：
- ...

注意：
- ...
```

如已实际检查文件，应说明：

- 入口文件存在；
- CSS / JS 路径可用，或已采用单文件内联；
- 页面板 Tab / 分组切换可用，或多页面主要链接存在；
- 页面状态归属符合材料语义；
- 未引入范围外功能。

---

## 12. 修改规则

当用户要求修 Bug 或改细节：

1. 先定位问题页面、选择器、数据入口或跳转入口；
2. 列出真实修改点；
3. 只改必要文件；
4. 不顺手重构；
5. 不引入框架；
6. 不把临时验证直接正式化；
7. 不因局部问题整包重写，除非用户明确要求重做。

修改后说明：

- 改了哪些文件；
- 解决了什么问题；
- 还有哪些未处理点。

---

## 13. 输出要求

默认输出：

- 一个可直接打开的实际原型预览链接；
- 简短说明；
- 本轮覆盖范围；
- 下一步建议。

不要输出压缩包。  
不要提示打包。  
不要生成图片参考图。  
不要在输出形态未确认时直接生成原型。  
不要把完整源码大段贴在聊天里，除非用户明确要求。
